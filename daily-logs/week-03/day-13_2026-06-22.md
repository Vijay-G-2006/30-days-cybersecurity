## Technical Log

### Part 1 — Understanding relational databases and SQL

**Step 1 — What is a relational database?**

A relational database organizes data into **tables** (also called relations) with **rows** and **columns**. Each table represents an entity (users, products, logs), and relationships between tables enable querying across multiple entities.

Key concepts:
- **Table** — a collection of related data organized in rows and columns
- **Row** — a single record in a table (e.g., one user)
- **Column** — an attribute or field (e.g., username, email, password)
- **Primary Key** — a unique identifier for each row (e.g., `user_id`)
- **Foreign Key** — a reference to a primary key in another table (enables relationships)
- **Schema** — the structure of a database (table names, columns, data types)

**Step 2 — What is SQL?**

SQL (Structured Query Language) is the standard language for interacting with relational databases. The four primary operations are:
- **SELECT** — retrieve data
- **INSERT** — add new data
- **UPDATE** — modify existing data
- **DELETE** — remove data

Additional operations include:
- **CREATE** / **DROP** — define or remove tables and databases
- **ALTER** — modify table structure
- **JOIN** — combine data from multiple tables
- **WHERE** / **GROUP BY** / **ORDER BY** — filter and organize results

### Part 2 — Lab setup and connecting to MySQL on Metasploitable 2

**Step 3 — Verify MySQL is running on the target**

From the host, I confirmed Metasploitable 2 was accessible and MySQL was listening:

```bash
# Check if MySQL port 3306 is open
nmap -p 3306 192.0.2.10
# Result: 3306/tcp open  mysql

# Verify from the terminal
netstat -tulpn | grep 3306
# Result: tcp  0  0  192.0.2.10:3306  0.0.0.0:*  LISTEN
```

**Step 4 — Connect to MySQL from the host**

MySQL is a client-server architecture. The `mysql` CLI client connects to the server:

```bash
# Basic connection (no authentication required in default Metasploitable)
mysql -h 192.0.2.10 -u root -p
# Prompted for password; default Metasploitable 2 often has empty or weak credentials
# For this lab: password was empty (just press Enter)
```

Once connected, the MySQL prompt appeared:

```
mysql>
```

This is the interactive MySQL command shell. All SQL commands end with a semicolon (`;`).

**Step 5 — Connecting directly on the Metasploitable VM**

For more direct access, I SSH'd into the Metasploitable 2 VM and connected locally:

```bash
# SSH into Metasploitable (login: msfadmin / password: msfadmin)
ssh msfadmin@192.0.2.10

# On the VM, connect to MySQL locally
mysql -u root
# (no password required on default installation)
```

Local connections do not require network transit and often have fewer restrictions.

### Part 3 — Basic database exploration and CRUD operations

**Step 6 — List databases and schema inspection**

Once connected to MySQL, I explored the existing databases:

```sql
-- List all databases
SHOW DATABASES;
-- Result:
-- information_schema
-- dvwa
-- mysql
-- test
-- wordpress

-- Select a database to work with
USE dvwa;

-- List tables in the current database
SHOW TABLES;
-- Result:
-- users
-- guestbook
-- messages

-- Inspect table structure
DESCRIBE users;
-- Result:
-- | Field    | Type        | Null | Key | Default | Extra          |
-- | user_id  | int(11)     | NO   | PRI | NULL    | auto_increment |
-- | username | varchar(15) | NO   | UNI | NULL    |                |
-- | password | varchar(32) | NO   |     | NULL    |                |
-- | email    | varchar(100)| NO   |     | NULL    |                |
-- | ...

-- Alternative: Show CREATE TABLE statement (reveals constraints)
SHOW CREATE TABLE users\G
```

> **OPSEC Note:** The `\G` terminator displays output vertically instead of horizontally, making large rows readable without truncation.

**Step 7 — SELECT queries: retrieving data**

The most common SQL operation is **SELECT**. Basic syntax:

```sql
SELECT column1, column2 FROM table WHERE condition;
```

Examples:

```sql
-- Retrieve all users
SELECT * FROM users;

-- Retrieve specific columns
SELECT username, email FROM users;

-- Filter with WHERE clause
SELECT username, email FROM users WHERE user_id = 1;

-- Count rows
SELECT COUNT(*) FROM users;
-- Result: 5 (if there are 5 users)

-- Filter by pattern matching
SELECT username FROM users WHERE username LIKE 'admin%';
-- LIKE with % (wildcard) matches any characters

-- Order results
SELECT username, email FROM users ORDER BY username ASC;

-- Limit results (useful for large tables)
SELECT username FROM users LIMIT 5;
```

**Step 8 — INSERT: adding new data**

```sql
-- Add a new user
INSERT INTO users (username, password, email) VALUES ('testuser', 'password123', 'test@example.com');

-- Verify insertion
SELECT * FROM users WHERE username = 'testuser';

-- Insert multiple rows
INSERT INTO users (username, password, email) VALUES 
  ('user2', 'pass2', 'user2@example.com'),
  ('user3', 'pass3', 'user3@example.com');
```

**Step 9 — UPDATE: modifying existing data**

```sql
-- Update a single field
UPDATE users SET password = 'newpassword' WHERE username = 'testuser';

-- Update multiple fields
UPDATE users SET password = 'reset', email = 'newemail@example.com' WHERE user_id = 5;

-- Be careful: without WHERE, ALL rows are updated
UPDATE users SET password = 'hacked';  -- DANGEROUS: updates all passwords
```

**Step 10 — DELETE: removing data**

```sql
-- Delete a specific user
DELETE FROM users WHERE username = 'testuser';

-- Delete all records (without WHERE clause)
DELETE FROM users;  -- DANGEROUS: removes all data

-- Safer deletion with verification
SELECT * FROM users WHERE user_id = 10;  -- Verify first
DELETE FROM users WHERE user_id = 10;    -- Then delete
```

> **Key lesson:** The absence of a WHERE clause in UPDATE or DELETE is catastrophic. Modern databases often require explicit confirmation or support transactions to recover from such mistakes.

### Part 4 — Advanced queries: JOINs, aggregation, and filtering

**Step 11 — JOIN: combining data from multiple tables**

Most real-world queries involve data from multiple tables. JOINs enable this:

```sql
-- First, understand the schema
DESCRIBE users;
DESCRIBE guestbook;

-- INNER JOIN: return matching rows from both tables
SELECT users.username, guestbook.comment 
FROM users 
INNER JOIN guestbook ON users.user_id = guestbook.user_id;

-- LEFT JOIN: return all rows from left table, matching rows from right
SELECT users.username, guestbook.comment 
FROM users 
LEFT JOIN guestbook ON users.user_id = guestbook.user_id;

-- Multiple JOINs
SELECT users.username, guestbook.comment, messages.text
FROM users
INNER JOIN guestbook ON users.user_id = guestbook.user_id
INNER JOIN messages ON users.user_id = messages.user_id;
```

**Step 12 — GROUP BY and aggregation**

```sql
-- Count messages per user
SELECT username, COUNT(*) as message_count
FROM users
LEFT JOIN messages ON users.user_id = messages.user_id
GROUP BY username;

-- Find users with more than 5 messages (HAVING filters groups)
SELECT username, COUNT(*) as message_count
FROM users
INNER JOIN messages ON users.user_id = messages.user_id
GROUP BY username
HAVING COUNT(*) > 5;

-- Other aggregate functions
SELECT 
  COUNT(*) as total_users,
  MAX(user_id) as highest_id,
  MIN(user_id) as lowest_id
FROM users;
```

**Step 13 — Subqueries**

```sql
-- Find users who have posted messages
SELECT username FROM users 
WHERE user_id IN (SELECT user_id FROM messages);

-- Find users with above-average message count
SELECT username FROM users
WHERE user_id IN (
  SELECT user_id FROM messages 
  GROUP BY user_id 
  HAVING COUNT(*) > (SELECT AVG(msg_count) FROM (
    SELECT COUNT(*) as msg_count FROM messages GROUP BY user_id
  ) as avg_table)
);
```

### Part 5 — Database administration and user management

**Step 14 — Create a new database and table**

```sql
-- Create a new database
CREATE DATABASE lab_db;

-- Switch to it
USE lab_db;

-- Create a table with constraints
CREATE TABLE employees (
  emp_id INT AUTO_INCREMENT PRIMARY KEY,
  first_name VARCHAR(50) NOT NULL,
  last_name VARCHAR(50) NOT NULL,
  email VARCHAR(100) UNIQUE,
  salary DECIMAL(10, 2),
  hire_date DATE,
  department_id INT,
  FOREIGN KEY (department_id) REFERENCES departments(dept_id)
);

-- Create a related table
CREATE TABLE departments (
  dept_id INT AUTO_INCREMENT PRIMARY KEY,
  dept_name VARCHAR(50) UNIQUE,
  location VARCHAR(100)
);
```

**Step 15 — User and permission management**

```sql
-- Create a new database user (limited privileges)
CREATE USER 'webapp'@'192.0.2.1' IDENTIFIED BY 'webapp_password';

-- Grant specific privileges
GRANT SELECT, INSERT, UPDATE ON lab_db.* TO 'webapp'@'192.0.2.1';

-- Verify grants
SHOW GRANTS FOR 'webapp'@'192.0.2.1';

-- Revoke privileges
REVOKE INSERT ON lab_db.* FROM 'webapp'@'192.0.2.1';

-- Change a user's password
ALTER USER 'webapp'@'192.0.2.1' IDENTIFIED BY 'new_password';

-- Delete a user
DROP USER 'webapp'@'192.0.2.1';
```

### Part 6 — Transactions and data integrity

**Step 16 — Understanding transactions**

Transactions are sequences of operations that either all succeed or all fail (ACID: Atomicity, Consistency, Isolation, Durability).

```sql
-- Start a transaction
START TRANSACTION;

-- Multiple operations
INSERT INTO users (username, password) VALUES ('trans_user', 'pass');
UPDATE users SET password = 'updated' WHERE username = 'trans_user';
DELETE FROM users WHERE user_id = 999;  -- Non-existent ID (no error)

-- Commit: make changes permanent
COMMIT;

-- Or: Rollback (undo all changes since START TRANSACTION)
-- ROLLBACK;
```

**Step 17 — Savepoints**

```sql
START TRANSACTION;

INSERT INTO users (username, password) VALUES ('user_a', 'pass_a');
SAVEPOINT sp1;

INSERT INTO users (username, password) VALUES ('user_b', 'pass_b');
SAVEPOINT sp2;

-- If an error occurs, roll back to sp2 (undo user_b only)
ROLLBACK TO sp2;

-- user_a is still inserted, user_b is not
COMMIT;
```

### Part 7 — SQL injection vulnerability recognition

**Step 18 — Understanding SQL injection**

SQL injection occurs when user input is concatenated into SQL queries without sanitization. Example vulnerable application code:

```php
// VULNERABLE CODE (never do this)
$username = $_GET['username'];
$query = "SELECT * FROM users WHERE username = '" . $username . "'";
$result = mysqli_query($connection, $query);
```

An attacker can inject SQL by providing a crafted input:

```
Input: ' OR '1'='1
Resulting query: SELECT * FROM users WHERE username = '' OR '1'='1'
Result: Returns ALL users (because '1'='1' is always true)
```

**Step 19 — Testing SQL injection in DVWA**

DVWA has a deliberately vulnerable SQL injection page. I tested it:

1. Navigated to `http://192.0.2.10/dvwa/vulnerabilities/sqli/`
2. In the "User ID" field, entered: `1' OR '1'='1`
3. Clicked "Submit"
4. **Result:** All users in the database were displayed

This demonstrated that the application directly concatenates user input into SQL queries.

**Step 20 — Safe SQL patterns: Prepared statements**

Modern applications prevent SQL injection using **prepared statements** (parameterized queries):

```php
// SAFE CODE: using prepared statements
$username = $_GET['username'];
$query = "SELECT * FROM users WHERE username = ?";
$stmt = $connection->prepare($query);
$stmt->bind_param("s", $username);
$stmt->execute();
$result = $stmt->get_result();
```

With prepared statements, the database distinguishes between query logic and user data. The input `' OR '1'='1` is treated as a literal string, not SQL code.

### Part 8 — Performance and security considerations

**Step 21 — Indexes and query performance**

Indexes speed up SELECT queries but slow down INSERT/UPDATE/DELETE (because the index must be updated). Common strategies:

```sql
-- Create an index on username (frequently searched)
CREATE INDEX idx_username ON users(username);

-- Composite index (search by multiple columns)
CREATE INDEX idx_user_email ON users(username, email);

-- View existing indexes
SHOW INDEX FROM users;

-- Drop an index
DROP INDEX idx_username ON users;

-- Use EXPLAIN to analyze query performance
EXPLAIN SELECT * FROM users WHERE username = 'admin';
-- Shows whether the index is used
```

**Step 22 — Security audit of the MySQL installation**

Default Metasploitable 2 MySQL has several vulnerabilities:

```sql
-- Check for anonymous users (security risk)
SELECT user, host FROM mysql.user;
-- Result: includes anonymous accounts ('' user with no password)

-- Check for users with overly permissive privileges
SHOW GRANTS FOR 'root'@'localhost';

-- Dangerous: test database accessible to all users
SHOW DATABASES;
-- Result: includes 'test' database with world-writable permissions
```

Proper hardening would:
1. Delete anonymous users
2. Remove default test databases
3. Disable remote root login
4. Enforce strong passwords
5. Limit user privileges to only required databases and operations

## Challenge Encountered

> **Error / Obstacle:** When first connecting to MySQL, I received `ERROR 1045 (28000): Access denied for user 'root'@'192.0.2.10'` when attempting to connect from the host.

**Root cause:** The default Metasploitable 2 configuration restricts root login from remote hosts for security (ironically, while leaving the database vulnerable to other attacks). Root can only login from localhost.

**Resolution:**
1. SSH'd into Metasploitable 2 directly
2. Connected to MySQL locally: `mysql -u root`
3. Created a database user with remote login permissions: `CREATE USER 'remote_user'@'%' IDENTIFIED BY 'password';`
4. Granted appropriate privileges
5. Connected from the host using the new user

> **Key realization:** Default configurations often have mixed security postures — some protections (localhost-only root) coexist with severe vulnerabilities (empty root password, weak application code).

## Key Takeaways

- **SQL is a lingua franca.** Every web application, data pipeline, and security tool that deals with structured data uses SQL. Understanding it is foundational.
- **Schema design drives security.** Proper use of constraints (PRIMARY KEY, FOREIGN KEY, UNIQUE, NOT NULL) prevents data corruption and aids in validation.
- **Prepared statements are non-negotiable.** SQL injection remains one of the OWASP Top 10 vulnerabilities because developers still concatenate user input into queries. Parameterized queries make injection impossible.
- **Indexes are a sharp tool.** They accelerate SELECT queries but degrade write performance and consume disk space. Index selection is a balancing act.
- **Transactions prevent inconsistency.** Multi-step operations that must all succeed or all fail are protected by transactions — essential for financial systems, user registration, inventory management.
- **Default configurations are security theater.** Metasploitable 2's MySQL has some protections (localhost-only root) but leaves critical vulnerabilities (weak passwords, SQL injection in web apps). Hardening requires systematic auditing.
