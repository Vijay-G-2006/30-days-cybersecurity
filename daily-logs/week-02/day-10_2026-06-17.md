## Technical Log

### Part 1 — DoS versus DDoS: definitions and attacker goals

**Step 1 — Define DoS and DDoS**

A **Denial of Service (DoS)** attack uses a single host or small number of hosts to overwhelm a target system so it becomes unavailable to legitimate users. A **Distributed Denial of Service (DDoS)** attack uses many compromised hosts or bots from across a network to amplify the volume and make the attack harder to block.

The two attack goals are:
- **Capacity exhaustion** — consume CPU, memory, socket table entries, or bandwidth
- **Service degradation** — force timeouts, authentication failures, or application errors

A local experiment is a safe way to learn the mechanics without harming external systems.

**Step 2 — Map attacks to OSI/Internet layers**

| Type | Layer | Example |
|---|---|---|
| Network/Transport | Layer 3/4 | SYN flood, UDP flood, ICMP flood |
| Application | Layer 7 | HTTP Slowloris, POST flood, expensive SQL queries |
| Amplification | Layer 3/4 | DNS/amplification, NTP reflection |

DoS is not just "more traffic" — it is *targeted traffic that exhausts a specific resource.*

### Part 2 — Building the local DoS lab

**Step 3 — Create a local target service**

Used a simple Python HTTP server as the victim service so I could safely observe resource exhaustion on my own machine.

```bash
mkdir -p ~/dos-lab
cd ~/dos-lab
python3 -m http.server 8080
```

The server started on `127.0.0.1:8080`. This is a clean, isolated service with no external exposure.

**Step 4 — Confirm the target is reachable**

In another terminal, verified the service and checked socket state:

```bash
curl -I http://127.0.0.1:8080
ss -tuln | grep 8080
```

The goal at this stage was to make sure the target accepted normal requests before launching any experiments.

### Part 3 — Layer 4 attacks: SYN flood and UDP flood with hping3

**Step 5 — SYN flood using hping3**

`hping3` can craft raw TCP packets with custom flags, sequence numbers, and source addresses. A SYN flood sends many SYN packets to open connections without ever completing the TCP handshake.

```bash
sudo hping3 --flood --rand-source -S -p 8080 127.0.0.1
```

- `--flood` sends packets as fast as possible
- `--rand-source` randomizes source IPs to simulate a distributed attack pattern
- `-S` sets the SYN flag
- `-p 8080` targets the local HTTP service

**Observation:** the local system rapidly filled the `SYN_RECV` connection table and the target service began dropping new packets.

Verified with:

```bash
ss -n state syn-recv
ss -s
```

**Step 6 — UDP flood with hping3**

A UDP flood targets a service or port with a stream of UDP packets. Because UDP is connectionless, the target may spend CPU and memory processing each incoming datagram.

```bash
sudo hping3 --flood --rand-source --udp -p 8080 127.0.0.1
```

While `python3 -m http.server` is not a UDP service, this exercise demonstrated how a flood can still saturate the host network stack and consume resources.

### Part 4 — Layer 7 attack: application-layer exhaustion with Slowloris

**Step 7 — Install and run Slowloris**

Slowloris is a Python tool that opens many HTTP connections to a web server and sends headers very slowly. It keeps sockets open, forcing the server to allocate resources until it can no longer accept valid clients.

```bash
# Install slowloris if not already present
python3 -m pip install --user slowloris

# Run against the local server
~/.local/bin/slowloris 127.0.0.1:8080 -s 100 -z 10
```

- `-s 100` opens 100 sockets
- `-z 10` sends a header line every 10 seconds

**Observation:** the local web server still responded normally to a small number of requests, but connection count remained high and new connections started receiving `Connection refused` or stalled.

**Step 8 — Differentiate application-layer and transport-layer attacks**

- **SYN flood** attacks the TCP handshake state and socket table.
- **Slowloris** attacks the HTTP server's request parser and worker threads while keeping connections half-open.
- The same server can be vulnerable to both if it maintains too many simultaneous sockets.

### Part 5 — Defensive controls and mitigation

**Step 9 — Enable TCP SYN cookies**

SYN cookies are a built-in kernel mitigation for SYN flood attacks. They allow the kernel to avoid allocating full connection state until the handshake is completed.

```bash
sudo sysctl -w net.ipv4.tcp_syncookies=1
sudo sysctl -w net.ipv4.tcp_max_syn_backlog=256
```

Verified the kernel setting:

```bash
sysctl net.ipv4.tcp_syncookies
```

**Step 10 — Add iptables rate limiting**

Rate limiting is a basic first line of defense for both SYN and HTTP flood attacks.

```bash
sudo iptables -A INPUT -p tcp --syn --dport 8080 -m limit --limit 10/s --limit-burst 20 -j ACCEPT
sudo iptables -A INPUT -p tcp --syn --dport 8080 -j DROP
```

This allows legitimate connection attempts while dropping excessive SYN packets.

**Step 11 — Test the defense**

Restarted the attack and watched the kernel and socket counters again. With SYN cookies enabled and iptables limit in place, the target service remained reachable for a small number of normal clients.

### Part 6 — Lessons from local experiments

**Step 12 — Why local attacks are useful**

Working in a local environment is essential for learning DoS safely. It allowed me to:
- see exactly how a flood changes socket state
- understand which resource is exhausted (handshake state vs request queue)
- validate that defenses like SYN cookies and rate limiting actually help

**Step 13 — What is different in a real DDoS?**

A real DDoS adds three complications:
1. **multiple attack sources** — makes IP-based blocking less effective
2. **amplification** — tiny requests generate large responses
3. **legitimate traffic mixing** — defenders must distinguish attack traffic from real users

A safe local experiment does not reproduce the botnet, but it does reproduce the same resource exhaustion patterns.

## Challenge Encountered

> **Error / Obstacle:** `hping3` required root privileges and the first attack accidentally disrupted my own desktop shell.

**Root cause:** `hping3` crafts raw packets, which the Linux kernel only allows for privileged users. The first flood attack also targeted `127.0.0.1`, so it affected the host machine immediately.

**Resolution:**
1. Stopped the attack with `Ctrl+C`.
2. Used `sudo` only for the packet-generation terminal.
3. Limited the flood duration and monitored `ss` continuously to avoid self-inflicted denial of service.

> **Key realization:** Doing DoS experiments on your own system requires careful control. A flood can take down the lab faster than the attacker can learn from it.

---

## Resolution

- Verified the target service before attack launch using `curl` and `ss`
- Used `sudo` only when needed for `hping3`
- Enabled TCP SYN cookies and rate limiting to preserve service availability
- Switched to application-layer testing with `slowloris` once the transport-layer experiment was complete

---

## Key Takeaways

- **DoS vs DDoS:** DoS comes from one source, DDoS comes from many. The defense mindset is the same: preserve resources and drop abusive traffic.
- **Layer matters:** A SYN flood attacks TCP state, while Slowloris attacks HTTP request handling. Protect both the transport layer and the application layer.
- **Local labs are safer:** Test only on isolated targets you own. Use localhost or a dedicated VM, not production systems.
- **Mitigations are practical:** Kernel settings like `tcp_syncookies` and firewall rate limits make a measurable difference under flood conditions.
- **Self-defense is planning:** When launching a DoS experiment on your own host, keep an escape plan and monitor socket state constantly.
