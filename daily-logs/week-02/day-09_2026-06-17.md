## Technical Log

### Part 1 — Understanding Phishing and Its Variants

**Step 1 — What is phishing?**

Phishing is a social engineering attack that tricks users into revealing sensitive information — credentials, financial data, personal details — by impersonating a trusted entity. Unlike technical exploits that target software vulnerabilities, phishing targets **human psychology**: trust, urgency, fear, and curiosity.

Phishing remains the #1 initial access vector in real-world breaches. According to industry reports, over 90% of successful cyberattacks begin with a phishing email. Understanding how these attacks work — from both the attacker's and defender's perspective — is essential for any cybersecurity professional.

**Step 2 — Types of phishing attacks**

| Type | Target | Method | Example |
|---|---|---|---|
| **Email Phishing** | Mass audience | Bulk emails with malicious links or attachments | "Your account has been compromised — click here to verify" |
| **Spear Phishing** | Specific individual | Personalized email using OSINT (name, role, company) | "Hi [Name], please review the attached Q2 report" |
| **Whaling** | C-suite executives | Highly targeted attack on senior leadership | Fake legal subpoena or urgent board communication |
| **Vishing** | Phone users | Voice call impersonating authority (bank, IT support) | "This is your bank — we detected suspicious activity" |
| **Smishing** | Mobile users | SMS messages with shortened malicious URLs | "Your package delivery failed — click to reschedule" |
| **Clone Phishing** | Previous email recipients | Re-send a legitimate email with replaced links/attachments | Identical email to a real one, but with a malicious attachment |
| **Pharming** | DNS-level attack | Redirect legitimate domain traffic to a fake server via DNS poisoning | Visiting `bank.com` resolves to attacker's IP |
| **Angler Phishing** | Social media users | Fake customer support accounts on Twitter/Instagram | "@bankSupport: We noticed your complaint — DM us your details" |
| **Evil Twin** | Wi-Fi users | Rogue access point mimicking legitimate Wi-Fi | Fake "Starbucks_WiFi" network with a captive portal harvesting credentials |
| **Watering Hole** | Targeted group | Compromise a website frequently visited by the target group | Inject malicious code into an industry-specific forum |

> **Key insight:** The sophistication of phishing doesn't lie in the technology — it lies in the social engineering. A well-crafted spear phishing email with proper context from OSINT is far more dangerous than a mass-blast generic email with obvious red flags.

### Part 2 — Credential Harvesting with Zphisher (Automated Approach)

**Step 3 — Installing and configuring Zphisher**

Zphisher is an automated phishing tool that provides pre-built templates for popular platforms (Google, Facebook, Instagram, Microsoft, etc.). It generates a local phishing server, serves the cloned page, and captures any credentials entered.

```bash
# Clone the Zphisher repository
git clone https://github.com/htr-tech/zphisher.git

# Navigate to the directory
cd zphisher

# Make the script executable
chmod +x zphisher.sh

# Launch Zphisher
bash zphisher.sh
```

On first run, Zphisher checks for and installs its dependencies (`php`, `curl`, `wget`). It then presents a menu of phishing templates:

```
[01] Facebook      [11] Twitch
[02] Instagram     [12] Pinterest
[03] Google        [13] Snapchat
[04] Microsoft     [14] LinkedIn
[05] Netflix       [15] Ebay
[06] PayPal        [16] Quora
[07] Steam         [17] Protonmail
[08] Twitter       [18] Spotify
[09] PlayStation   [19] WordPress
[10] GitHub        [20] Origin
...
```

**Step 4 — Launching a phishing page on localhost**

Selected Google as the target template to understand the workflow:

```
[?] Select an option: 03
[?] Select a port [default: 8080]: 8080
```

Zphisher starts a local PHP server on `127.0.0.1:8080`, serving a pixel-perfect clone of Google's login page. Opening `http://127.0.0.1:8080` in a browser presents a page visually identical to `accounts.google.com`.

When credentials are entered on the fake page:

```
[*] Victim has opened the phishing page
[*] IP Address found: 127.0.0.1
[*] Credentials captured:
    Email: victim@example.com
    Password: examplepass123
```

The credentials are logged to a file (`auth/usernames.dat`) for review. The victim is then redirected to the legitimate Google login page — making the attack seamless and less suspicious.

> **OPSEC Note:** All credentials shown are test values entered by the attacker (self) in a controlled lab. No real credentials were harvested from any user.

**Step 5 — Hosting the phishing page externally via tunneling**

A localhost-only phishing page is limited to the local network. To simulate a real-world attack where the victim accesses the page from the internet, Zphisher integrates with tunneling services:

```
[?] Choose a tunneling option:
[01] Localhost only
[02] Ngrok (requires auth token)
[03] Cloudflared
[04] LocalXpose
```

Selecting Ngrok (after configuring an auth token):

```bash
# Authenticate Ngrok (one-time setup)
ngrok config add-authtoken <YOUR_AUTH_TOKEN>
```

Zphisher automatically launches Ngrok and provides a public URL:

```
[*] Ngrok URL: https://xxxx-xx-xx-xxx-xx.ngrok-free.app
[*] Send this URL to the victim
```

This URL serves the exact same cloned page but is now accessible from anywhere on the internet. Any credentials entered are still captured on the attacker's local machine.

> **Critical distinction:** The localhost test proves the concept; the tunneled test proves the **delivery mechanism**. In a real engagement, the URL would be embedded in a phishing email with a convincing pretext.

### Part 3 — Credential Harvesting with SET (Social Engineering Toolkit)

**Step 6 — Introduction to SET**

The Social Engineering Toolkit (SET) is a far more powerful and flexible framework than Zphisher. Created by David Kennedy (TrustedSec), SET is designed for advanced social engineering penetration testing. While Zphisher provides templates, SET can clone **any** website in real-time and provides additional attack vectors (email spoofing, payload generation, QR code attacks).

```bash
# Launch SET (requires root)
sudo setoolkit
```

SET presents a modular menu system:

```
1) Social-Engineering Attacks
2) Penetration Testing (Fast-Track)
3) Third Party Modules
4) Update the Social-Engineering Toolkit
5) Update SET configuration
99) Exit the Social-Engineering Toolkit
```

**Step 7 — Website cloning and credential harvesting with SET**

Navigated through the menus to reach the credential harvester:

```
Select: 1  → Social-Engineering Attacks
Select: 2  → Website Attack Vectors
Select: 3  → Credential Harvester Attack Method
Select: 2  → Site Cloner
```

SET then prompts for the attacker's IP (where the cloned page will be served) and the URL to clone:

```
set:webattack> IP address for the POST back in Harvester/Tabnabbing [192.0.2.10]:
[-] SET supports both HTTP and HTTPS
set:webattack> Enter the url to clone: https://accounts.google.com
[*] Cloning the website: https://accounts.google.com
[*] This could take a moment...
[*] Site has been successfully cloned
[*] The Social-Engineer Toolkit Credential Harvester Attack
[*] Credential Harvester is running on port 80
[*] Information will be displayed to you as it arrives below:
```

Unlike Zphisher's static templates, SET performs a **live clone** of the target URL — downloading the HTML, CSS, JavaScript, and images in real-time. This means it captures the **current** version of any website, not a potentially outdated template.

**Step 8 — Capturing credentials via SET**

When a victim visits the attacker's IP and submits credentials on the cloned page:

```
[*] WE GOT A HIT! Printing the output:
PARAM: email=victim@example.com
PARAM: Passwd=examplepass123
PARAM: PersistentCookie=yes
[*] POSSIBLE USERNAME FIELD FOUND: victim@example.com
[*] POSSIBLE PASSWORD FIELD FOUND: examplepass123
[*] The captured credentials have been saved to:
    /root/.set/reports/
```

SET automatically parses POST parameters, identifies likely username and password fields, and saves a full report in XML and HTML format under `/root/.set/reports/`.

> **Advantage of SET over Zphisher:** SET clones any arbitrary URL live, captures all POST parameters (not just username/password), generates formatted reports, and integrates with other SET modules (email spoofing, Metasploit payloads). Zphisher is faster for quick demos, but SET is the professional-grade tool.

**Step 9 — Comparing Zphisher vs. SET**

| Feature | Zphisher | SET (Social Engineering Toolkit) |
|---|---|---|
| **Ease of use** | Very easy — menu-driven, beginner-friendly | Moderate — requires understanding of attack flow |
| **Templates** | Pre-built templates for ~30+ popular sites | No templates — clones any URL live |
| **Website cloning** | Static templates (may be outdated) | Real-time live cloning of any target URL |
| **Credential capture** | Username and password only | All POST parameters (tokens, cookies, hidden fields) |
| **Tunneling** | Built-in Ngrok, Cloudflared, LocalXpose | Manual setup required (Ngrok, port forwarding) |
| **Reporting** | Basic text file (`auth/usernames.dat`) | Formatted reports in XML/HTML (`/root/.set/reports/`) |
| **Integration** | Standalone tool | Integrates with Metasploit, email spoofing, payloads |
| **Best for** | Quick demonstrations and learning | Professional penetration testing engagements |

### Part 4 — Hosting Phishing Pages on External Servers

**Step 10 — Understanding hosting mechanisms for phishing pages**

In real-world phishing campaigns, attackers use various mechanisms to host and deliver their cloned pages:

| Mechanism | How It Works | Detection Difficulty |
|---|---|---|
| **Local + Tunnel (Ngrok/Cloudflared)** | Run server locally, expose via tunnel URL | Moderate — tunnel domains are increasingly flagged |
| **VPS Hosting** | Deploy phishing page on a rented VPS with a custom domain | High — custom domains with valid SSL look legitimate |
| **Compromised Websites** | Inject phishing page into a legitimate but compromised website | Very High — the domain already has a good reputation |
| **Cloud Functions (Serverless)** | Host credential harvester as a serverless function | High — uses trusted cloud provider domains |
| **URL Shorteners** | Mask the phishing URL behind bit.ly, tinyurl, etc. | Low — but bypasses visual URL inspection |
| **Data URIs / JavaScript** | Encode the entire phishing page in the URL itself | Moderate — no external hosting needed |
| **IDN Homograph Attack** | Register domains using Unicode characters that look like ASCII (e.g., `gооgle.com` with Cyrillic "о") | Very High — visually identical to legitimate domain |

**Step 11 — Testing with Ngrok (hosted environment)**

Configured SET to work with Ngrok for external hosting:

```bash
# Terminal 1: Start SET credential harvester on port 80
sudo setoolkit
# Navigate: 1 → 2 → 3 → 2 → enter IP → enter URL to clone

# Terminal 2: Expose port 80 via Ngrok
ngrok http 80
```

Ngrok provides a forwarding URL:

```
Forwarding    https://xxxx-xx-xx-xxx-xx.ngrok-free.app → http://localhost:80
```

Accessing this URL from any device (phone, another computer) loads the cloned page. Credentials entered are captured by SET running locally. The entire attack chain works end-to-end:

1. Attacker clones target site with SET
2. Ngrok exposes the local server to the internet
3. Victim receives the Ngrok URL (via phishing email, SMS, etc.)
4. Victim enters credentials on the cloned page
5. SET captures and logs all POST parameters
6. Victim is redirected to the real website (no suspicion raised)

### Part 6 — Phishing Defense and Detection Mechanisms

**Step 13 — How to identify phishing attempts**

From a defender's perspective, the following indicators help identify phishing:

| Indicator | What to Check | Example Red Flag |
|---|---|---|
| **Sender email** | Domain mismatch | `support@g00gle-security.com` instead of `@google.com` |
| **URL inspection** | Hover over links before clicking | URL shows `login.google.com.attacker.xyz` |
| **SSL certificate** | Check certificate issuer in browser | Free/self-signed cert on a "bank" website |
| **Urgency language** | "Act immediately" / "Account suspended" | Pressure to bypass rational thinking |
| **Generic greeting** | "Dear Customer" instead of your name | Mass phishing, not personalized |
| **Attachments** | Unexpected file types | `.exe`, `.scr`, `.js` disguised as invoices |
| **Grammar/formatting** | Poor language, inconsistent branding | Obvious when compared to legitimate communications |

**Step 14 — Technical defenses against phishing**

| Defense | Layer | How It Works |
|---|---|---|
| **SPF (Sender Policy Framework)** | Email | DNS record specifying which IPs can send email for a domain |
| **DKIM (DomainKeys Identified Mail)** | Email | Cryptographic signature in email headers proving the email wasn't modified in transit |
| **DMARC (Domain-based Message Authentication)** | Email | Policy combining SPF and DKIM to instruct receiving servers on how to handle failures |
| **Multi-Factor Authentication (MFA)** | Account | Even if credentials are phished, the attacker needs the second factor |
| **Security Awareness Training** | Human | Regular phishing simulations and training for employees |
| **URL Filtering / Web Proxy** | Network | Block access to known phishing domains and newly registered domains |
| **Email Gateway Filtering** | Email | Scan incoming emails for malicious links, attachments, and sender reputation |
| **Browser Phishing Protection** | Endpoint | Built-in browser warnings (Google Safe Browsing, SmartScreen) |
| **FIDO2/WebAuthn (Passkeys)** | Account | Phishing-resistant authentication — cryptographic keys bound to the legitimate domain |

> **The ultimate defense:** FIDO2/WebAuthn (Passkeys) is the only authentication mechanism that is fundamentally **phishing-proof**. Because the cryptographic key is bound to the legitimate domain, it simply won't authenticate on a cloned phishing page — even if the victim visits it. MFA via SMS or TOTP can still be phished using real-time relay proxies (tools like Evilginx2), but hardware keys and passkeys cannot.

## Challenge Encountered

> **Obstacle:** When running SET's credential harvester, the cloned website initially failed to load CSS and images — the phishing page appeared as raw unstyled HTML, which would immediately raise suspicion from any victim. The page was functional (forms worked) but visually unconvincing.

**Root cause:** SET clones the HTML but some externally referenced assets (CSS stylesheets, JavaScript files, images hosted on CDNs) are loaded via absolute URLs pointing to the original domain. Modern websites use Content Security Policy (CSP) headers and cross-origin restrictions that prevent these assets from loading when served from a different origin (the attacker's IP/domain).

---

## Resolution

1. Identified the missing assets by inspecting the browser's Developer Console (F12 → Console tab) — multiple `ERR_BLOCKED_BY_CSP` and `404 Not Found` errors appeared
2. For the SET approach: used the `--apache` flag alternative to serve through Apache with proper headers, which improved asset loading
3. For Zphisher: the pre-built templates already include all assets locally (CSS, images, fonts are embedded), which is why Zphisher pages render correctly out-of-the-box despite being less flexible
4. For maximum fidelity with SET: manually downloaded critical CSS and image assets, modified the cloned HTML to reference local copies instead of CDN URLs

> **Key realization:** The visual fidelity of a phishing page is directly proportional to its success rate. A pixel-perfect clone is exponentially more effective than a broken-layout page. This is why Zphisher's static templates (despite being potentially outdated) often look better than SET's live clones — they include all assets locally. Professional phishing engagements require manual asset embedding to achieve both accuracy and visual fidelity.

## Key Takeaways

- **Phishing targets people, not systems.** Unlike exploits that abuse software vulnerabilities, phishing abuses human psychology — trust, urgency, fear, and authority. No firewall can patch a human clicking a link because they believe their account is compromised.
- **Credential harvesting is trivially easy.** Tools like Zphisher and SET can clone any login page and capture credentials in minutes. The barrier to entry for launching a phishing attack is alarmingly low, which is precisely why it remains the most common attack vector worldwide.
- **MFA is necessary but not sufficient.** While multi-factor authentication blocks most credential reuse attacks, advanced phishing frameworks (Evilginx2, Modlishka) can relay MFA tokens in real-time. Only phishing-resistant authentication (FIDO2/WebAuthn/Passkeys) provides fundamental protection by cryptographically binding authentication to the legitimate domain.
