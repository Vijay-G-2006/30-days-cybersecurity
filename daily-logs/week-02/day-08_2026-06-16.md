## Technical Log

### Part 1 — Proper Installation and Configuration

**Step 1 — Installing Wireshark and resolving the permissions model**

On most Linux distributions, raw packet capture requires root privileges because it accesses the network interface at the data-link layer. However, running a GUI application like Wireshark as root is a security risk — any vulnerability in Wireshark's protocol dissectors could be exploited with full root privileges.

The correct approach is to add the user to the `wireshark` group, which grants access to the capture interface without full root:

```bash
# Install Wireshark (Arch Linux)
sudo pacman -S wireshark-qt

# During installation, the package creates a 'wireshark' group
# Add the current user to it
sudo usermod -aG wireshark $USER
```

> **Critical:** The group membership change does not take effect until the user logs out and back in (or starts a new login shell). Running `groups` immediately after `usermod` will NOT show the new group — this is a common source of confusion.

```bash
# Verify the group was added (after re-login)
groups
# Output should include: wireshark

# Launch Wireshark as a normal user (no sudo)
wireshark &
```

**Step 2 — Reconfiguring the package if capture permissions fail**

If Wireshark still cannot access network interfaces after adding the user to the group, the dumpcap binary may not have the correct capabilities set. This is resolved by reconfiguring the package:

```bash
# Check if dumpcap has the required capability
getcap /usr/bin/dumpcap
# Expected: /usr/bin/dumpcap cap_net_raw,cap_net_admin=eip

# If missing, reconfigure (Arch Linux)
sudo chmod 750 /usr/bin/dumpcap
sudo chgrp wireshark /usr/bin/dumpcap
sudo setcap cap_net_raw,cap_net_admin=eip /usr/bin/dumpcap
```

On Debian/Ubuntu-based systems, the equivalent is:

```bash
sudo dpkg-reconfigure wireshark-common
# Select "Yes" when asked to allow non-superusers to capture packets
```

**Step 3 — Understanding Wireshark's interface layout**

After launching, selected the active network interface (e.g., `eth0`, `wlan0`, or `enp0s3`) to begin capturing. The main Wireshark window has three panes:

| Pane | Location | What It Shows |
|---|---|---|
| **Packet List** | Top | One row per captured packet — number, timestamp, source IP, destination IP, protocol, length, and a summary |
| **Packet Details** | Middle | Expandable, layer-by-layer breakdown of the selected packet (Ethernet → IP → TCP/UDP → Application) |
| **Packet Bytes** | Bottom | Raw hexadecimal and ASCII dump of the packet's bytes |

The Packet Details pane is the most powerful — it decodes every layer of the protocol stack, exposing headers, flags, sequence numbers, and payload data in human-readable form.

### Part 2 — Capture Filters vs. Display Filters

Wireshark has two distinct filter systems, and confusing them is one of the most common beginner mistakes.

**Step 4 — Capture filters (BPF syntax — applied BEFORE capture)**

Capture filters use Berkeley Packet Filter (BPF) syntax and are set before starting a capture. They determine which packets Wireshark even records. Packets that don't match the capture filter are discarded entirely — they cannot be recovered.

```
# Capture filter examples (set in Wireshark's capture options dialog)

host 192.0.2.10               # Only capture traffic to/from this IP
port 80                       # Only capture traffic on port 80
net 192.0.2.0/24              # Only capture traffic on this subnet
tcp                           # Only capture TCP packets
udp                           # Only capture UDP packets
not arp                       # Capture everything except ARP
host 192.0.2.10 and port 443  # Combine: specific host AND specific port
```

> **When to use capture filters:** On busy networks where capturing everything would produce gigabytes of data. Capture filters reduce the volume at the source, keeping the capture file manageable.

**Step 5 — Display filters (Wireshark syntax — applied AFTER capture)**

Display filters use Wireshark's own filter syntax and are applied to packets that have already been captured. They control what is *shown* in the Packet List pane without discarding anything — the full capture remains in memory.

```
# Display filter examples (typed into the filter bar at the top of Wireshark)

ip.addr == 192.0.2.10                     # All traffic to/from this IP
ip.src == 192.0.2.10                      # Only traffic FROM this IP (source)
ip.dst == 192.0.2.10                      # Only traffic TO this IP (destination)
tcp.port == 80                            # TCP traffic on port 80 (source or dest)
tcp.dstport == 443                        # TCP traffic destined for port 443
udp.port == 53                            # DNS traffic (UDP port 53)
```

| Feature | Capture Filter | Display Filter |
|---|---|---|
| **Syntax** | BPF (`host`, `port`, `net`) | Wireshark (`ip.addr`, `tcp.port`) |
| **Applied** | Before capture starts | After packets are captured |
| **Effect** | Discards non-matching packets permanently | Hides non-matching packets (still in memory) |
| **Use case** | Reducing capture volume on busy networks | Focused analysis of already-captured traffic |

### Part 3 — Advanced Display Filters (Doing Nmap's Job with Wireshark)

This was the most technically dense part of the day — building Wireshark filters that replicate the reconnaissance capabilities of Nmap, but from passively captured traffic rather than active scanning.

**Step 6 — Finding open ports (equivalent to Nmap port scan)**

Nmap actively sends SYN packets and waits for SYN-ACK responses to determine open ports. In Wireshark, the same information can be extracted passively from already-captured traffic:

```
# Find all SYN-ACK packets — these reveal OPEN ports on responding hosts
tcp.flags.syn == 1 && tcp.flags.ack == 1

# Find all RST packets — these reveal CLOSED ports
tcp.flags.reset == 1

# Find SYN packets that received no response — potentially FILTERED ports
tcp.flags.syn == 1 && tcp.flags.ack == 0
```

By examining SYN-ACK packets, the source IP reveals the server and `tcp.srcport` reveals the open port — identical information to an Nmap scan, but derived from passive observation.

**Step 7 — Service and version detection (equivalent to Nmap -sV)**

Nmap's `-sV` flag sends protocol-specific probes and matches responses against a database. Wireshark achieves this differently — it dissects the actual application-layer data:

```
# Filter by protocol — Wireshark automatically identifies protocols
http                                      # All HTTP traffic
dns                                       # All DNS traffic
ssh                                       # All SSH traffic
ftp                                       # All FTP traffic
tls                                       # All TLS/SSL traffic
smtp                                      # All SMTP (email) traffic
smb                                       # All SMB (file sharing) traffic

# Find SSH version banners
ssh.protocol                              # Packets containing SSH version string
# In the Packet Details: "SSH Protocol: SSH-2.0-OpenSSH_8.9p1"

# Find HTTP server versions
http.server                               # Packets containing Server header
# In the Packet Details: "Server: Apache/2.4.54 (Ubuntu)"

# Find TLS/SSL certificate information
tls.handshake.type == 11                  # Certificate messages
# Exposes: certificate issuer, subject, validity dates, cipher suites
```

> **Key advantage over Nmap:** Wireshark doesn't just tell you "port 80 is open and running Apache" — it shows you the actual HTTP conversation, including every header, cookie, and response body. It provides the **full context** that Nmap summarizes in one line.

**Step 8 — OS fingerprinting (equivalent to Nmap -O)**

While not as precise as Nmap's dedicated OS fingerprinting, several TCP/IP characteristics visible in Wireshark can hint at the operating system:

```
# Filter for initial SYN packets (connection establishment)
tcp.flags.syn == 1 && tcp.flags.ack == 0
```

In the Packet Details pane, examine:

| TCP Field | Windows | Linux | macOS |
|---|---|---|---|
| **TTL (Time to Live)** | 128 | 64 | 64 |
| **Window Size** | 65535 | 29200 or 64240 | 65535 |
| **Don't Fragment (DF)** | Set | Set | Set |
| **TCP Options** | MSS, NOP, Window Scale, NOP, NOP, SACK | MSS, SACK, Timestamp, NOP, Window Scale | MSS, NOP, Window Scale, NOP, NOP, Timestamp, SACK |

A packet with TTL=128 and Window Size=65535 strongly suggests Windows. TTL=64 with Window Size=29200 suggests Linux.

### Part 4 — Extracting Sensitive Data from Captured Traffic

**Step 9 — Capturing cleartext credentials (HTTP, FTP, Telnet)**

Any protocol that transmits data without encryption is completely visible in Wireshark. This is the most powerful demonstration of why unencrypted protocols are dangerous.

**HTTP POST credentials (login forms over HTTP):**
```
# Filter for HTTP POST requests (login form submissions)
http.request.method == "POST"
```

Selecting a POST request to a login page and expanding the **"HTML Form URL Encoded"** section in Packet Details reveals the form fields in cleartext:

```
Form item: "username" = "admin"
Form item: "password" = "secret123"
```

> **OPSEC Note:** The credentials shown above are sanitized examples from the Metasploitable 2 lab, not real credentials. This demonstrates why HTTP (not HTTPS) login pages are a critical vulnerability.

**FTP credentials:**
```
# Filter for FTP commands
ftp.request.command == "USER" || ftp.request.command == "PASS"
```

FTP transmits usernames and passwords in complete cleartext:
```
Request: USER admin
Request: PASS password123
Response: 230 Login successful
```

**Telnet sessions:**
```
# Filter for Telnet traffic
telnet
```

Telnet transmits the entire session — including typed commands and their output — in cleartext, character by character. Wireshark can reconstruct the full session by right-clicking and selecting **"Follow → TCP Stream"**.

**Step 10 — Following TCP streams (reconstructing conversations)**

The **"Follow TCP Stream"** feature is one of Wireshark's most powerful tools. It reconstructs the entire conversation between a client and server from individual packets into a readable dialogue:

1. Right-click any packet in a conversation
2. Select **"Follow → TCP Stream"**

The result shows client data in one color (typically red) and server data in another (typically blue), presenting the entire exchange as a continuous conversation. This works for HTTP, FTP, Telnet, SMTP, and any other TCP-based protocol.

```
# Alternative: filter to show only packets in a specific TCP stream
tcp.stream eq 5                           # Show only TCP stream #5
```

**Step 11 — DNS analysis (mapping the target's behavior)**

DNS queries reveal which domains a host is communicating with — extremely valuable for behavioral analysis:

```
# All DNS queries (requests)
dns.flags.response == 0

# All DNS responses
dns.flags.response == 1

# Find queries for a specific domain
dns.qry.name contains "example"

# Find DNS errors (NXDOMAIN — domain doesn't exist)
dns.flags.rcode == 3
```

DNS traffic reveals:
- What websites and services a host is connecting to
- If a host is communicating with known malicious domains (C2 beaconing)
- DNS tunneling attempts (unusually long subdomain labels)

### Part 5 — Network Anomaly Detection

**Step 12 — Detecting port scans**

A port scan is visible in Wireshark as a rapid sequence of SYN packets from a single source to many different destination ports:

```
# SYN packets without ACK (initial connection attempts)
tcp.flags.syn == 1 && tcp.flags.ack == 0

# Filter for a specific suspicious source
ip.src == 192.0.2.50 && tcp.flags.syn == 1 && tcp.flags.ack == 0
```

In the Packet List, a port scan appears as hundreds of SYN packets in rapid succession, each targeting a different destination port. The Wireshark **"Statistics → Conversations"** view shows the scanning host communicating with an unusually high number of distinct ports.

**Step 13 — Detecting ARP spoofing**

ARP spoofing (used in MITM attacks) generates duplicate ARP responses. Wireshark has built-in detection:

```
# Filter for ARP traffic
arp

# Wireshark's Expert Info will flag:
# "Duplicate IP address detected" — a strong indicator of ARP spoofing
```

Navigate to **"Analyze → Expert Information"** to see Wireshark's automated warnings about protocol anomalies, including ARP duplicates, TCP retransmissions, and malformed packets.

**Step 14 — Statistical analysis for anomaly detection**

Wireshark provides several statistical views beyond the raw packet list:

| Menu Path | What It Shows |
|---|---|
| **Statistics → Protocol Hierarchy** | Percentage breakdown of all protocols — an unexpected protocol (e.g., IRC on a corporate network) is an anomaly |
| **Statistics → Conversations** | Every unique IP pair and their data exchange volume — identifies the heaviest communicators |
| **Statistics → Endpoints** | Every unique IP or MAC address seen on the network |
| **Statistics → I/O Graphs** | Traffic volume over time — spikes may indicate data exfiltration or DDoS |
| **Statistics → DNS** | All DNS queries with response counts — high NXDOMAIN rates suggest malware or tunneling |

### Part 6 — tshark: Wireshark on the Command Line

**Step 15 — Using tshark for headless packet analysis**

`tshark` is Wireshark's CLI counterpart — it uses the same dissectors and display filter syntax but runs in the terminal. This is essential for analyzing captures on remote servers without a GUI.

```bash
# Capture 100 packets on interface eth0
tshark -i eth0 -c 100

# Capture with a display filter (only HTTP traffic)
tshark -i eth0 -Y "http"

# Read a saved capture file and apply a filter
tshark -r capture.pcapng -Y "ip.addr == 192.0.2.10"

# Extract specific fields (like Nmap -sV output)
tshark -r capture.pcapng -Y "tcp.flags.syn==1 && tcp.flags.ack==1" \
  -T fields -e ip.src -e tcp.srcport

# Export HTTP objects (files transferred over HTTP)
tshark -r capture.pcapng --export-objects http,/tmp/exported_files/

# Count packets per protocol
tshark -r capture.pcapng -q -z io,phs
```

> **Key advantage of tshark:** It can be piped into other Unix tools (`grep`, `awk`, `sort`, `uniq`), enabling automated analysis workflows that are impossible with the GUI.

## Challenge Encountered

> **Obstacle:** After installing Wireshark and launching it as a regular user, no network interfaces appeared in the interface list. The capture area was completely empty, showing the message "No interfaces found."

**Root cause:** The user had not been added to the `wireshark` group. Without group membership, the user had no permission to access the `dumpcap` binary (which performs the actual raw packet capture). Additionally, the `dumpcap` binary lacked the `cap_net_raw` and `cap_net_admin` Linux capabilities that allow non-root processes to capture raw network frames.

---

## Resolution

1. Added the user to the `wireshark` group: `sudo usermod -aG wireshark $USER`
2. Logged out and back in for the group change to take effect (verified with `groups`)
3. Verified that `dumpcap` had the correct capabilities: `getcap /usr/bin/dumpcap`
4. When capabilities were missing, applied them manually: `sudo setcap cap_net_raw,cap_net_admin=eip /usr/bin/dumpcap`
5. Relaunched Wireshark — all network interfaces now appeared and capture worked correctly

> **Key realization:** Linux capabilities (`cap_net_raw`, `cap_net_admin`) are a fine-grained alternative to running as root. They grant specific kernel-level permissions to individual binaries, following the principle of least privilege. Understanding this system is essential for configuring security tools that need low-level network access without requiring full root.

## Key Takeaways

- **Wireshark is passive Nmap.** While Nmap actively probes targets by sending crafted packets, Wireshark achieves similar reconnaissance (open ports, service versions, OS hints) by passively analyzing traffic already flowing on the network — making it invisible to IDS systems.
- **Capture filters and display filters are fundamentally different.** Capture filters (BPF) discard packets before recording; display filters (Wireshark syntax) hide packets in the UI without deleting them. Using the wrong one at the wrong time either loses data permanently or fails to reduce capture volume.
- **Unencrypted protocols expose everything.** HTTP, FTP, and Telnet transmit credentials, commands, and data in complete cleartext. A single Wireshark capture on a network using these protocols reveals usernames, passwords, session tokens, and full conversation content — the strongest possible argument for HTTPS and SSH everywhere.
