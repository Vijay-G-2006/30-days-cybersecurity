# Day 7 — Web Application Interception and Attack Automation with Burp Suite

> *Set up a vulnerable lab environment with Metasploitable 2, configured Burp Suite as an intercepting proxy, and explored its core modules — Proxy, Repeater, Intruder — to understand how web requests can be intercepted, modified, replayed, and brute-forced.*

## Technical Log

### Part 1 — Lab Setup

**Step 1 — Deploying Metasploitable 2 as the target**

Metasploitable 2 is a deliberately vulnerable Linux virtual machine designed by Rapid7 for practicing penetration testing. It ships with intentionally misconfigured services, weak credentials, and known-vulnerable web applications (DVWA, Mutillidae, TWiki, etc.).

```bash
# Download the Metasploitable 2 VM image
# (Pre-built .vmdk file, no installation required)

# After importing into VirtualBox:
# 1. Set network adapter to "Host-Only" or "Internal Network"
#    — ensures the vulnerable VM is NOT exposed to the internet
# 2. Boot the VM
# 3. Login with default credentials: msfadmin / msfadmin
```

After booting, identified the VM's IP address:

```bash
# On the Metasploitable 2 VM
ifconfig
# Result: 192.0.2.10 (sanitized — actual lab IP noted privately)
```

> **OPSEC Note:** Metasploitable 2 must NEVER be connected to a bridged or NAT network with internet access. It is intentionally riddled with vulnerabilities and would be compromised within minutes on an open network. Host-Only networking isolates it to communication only between the host and the VM.

**Step 2 — Verifying the target is reachable**

From the host machine, confirmed connectivity and identified running web services:

```bash
# Ping the target
ping -c 4 192.0.2.10

# Quick Nmap scan to confirm web services
nmap -sV -p 80,443,8080,8180 192.0.2.10
# Result: Port 80 open — Apache httpd 2.2.8
#         Port 8180 open — Apache Tomcat/Coyote JSP engine 1.1
```

Navigating to `http://192.0.2.10` in the browser displayed the Metasploitable 2 landing page with links to all the vulnerable web applications.

**Step 3 — Installing and configuring Burp Suite**

Burp Suite is a Java-based web application security testing platform. The Community Edition is free and includes the essential tools: Proxy, Repeater, Intruder (rate-limited), Decoder, and Comparer.

```bash
# On Arch Linux, Burp Suite can be installed via AUR or downloaded from PortSwigger
# Launch Burp Suite
burpsuite &
```

On launch, Burp Suite starts a local proxy listener on `127.0.0.1:8080` by default.

**Step 4 — Configuring Firefox to route traffic through Burp**

For Burp Suite to intercept browser traffic, the browser must be configured to send all HTTP/HTTPS requests through Burp's proxy:

1. Open Firefox → Settings → Network Settings → Manual Proxy Configuration
2. Set HTTP Proxy: `127.0.0.1`, Port: `8080`
3. Check "Also use this proxy for HTTPS"
4. Remove `localhost, 127.0.0.1` from the "No Proxy for" field (to intercept local traffic too, if needed)

Alternatively, use the FoxyProxy browser extension for quick proxy toggling.

> **How it works:** With this configuration, every HTTP/HTTPS request from Firefox is first sent to Burp Suite (sitting at `127.0.0.1:8080`) before being forwarded to the actual target. This gives Burp full visibility and control over every request and response — the core of web application testing.

**Step 5 — Installing Burp's CA certificate (for HTTPS interception)**

For HTTPS sites, Burp Suite performs a man-in-the-middle (MITM) operation. It generates its own SSL certificate for each target domain on the fly. The browser must trust Burp's Certificate Authority (CA) to avoid certificate errors:

1. With the proxy active, navigate to `http://burp` in Firefox
2. Download the CA certificate (`cacert.der`)
3. Import into Firefox: Settings → Privacy & Security → Certificates → View Certificates → Import

### Part 2 — Burp Suite Proxy: Intercepting and Modifying Live Traffic

**Step 6 — Intercepting requests in real time**

The **Proxy** module is Burp's core — it is the intercepting proxy that sits between the browser and the target server.

1. In Burp Suite, navigated to **Proxy → Intercept** tab
2. Ensured **"Intercept is on"** was toggled active
3. In Firefox, navigated to `http://192.0.2.10/dvwa/login.php` (DVWA — Damn Vulnerable Web Application)

The browser appeared to hang — this is expected. Burp had caught the outgoing HTTP request and was holding it for inspection before forwarding.

**What the intercepted request looked like:**

```http
GET /dvwa/login.php HTTP/1.1
Host: 192.0.2.10
User-Agent: Mozilla/5.0 (X11; Linux x86_64; rv:128.0) Gecko/20100101 Firefox/128.0
Accept: text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8
Accept-Language: en-US,en;q=0.5
Accept-Encoding: gzip, deflate
Connection: keep-alive
```

Key observations from a single intercepted request:
- **Host header** — reveals the target server
- **User-Agent** — reveals the browser and OS of the client
- **Accept headers** — reveal what content types the client expects
- **Cookies** — if present, reveal session tokens (critical for session hijacking)

**Step 7 — Modifying requests before forwarding**

The real power of interception is the ability to **edit the request** before it reaches the server. This enables testing for:
- **Parameter tampering** — changing form values, IDs, prices
- **Header manipulation** — modifying User-Agent, Referer, cookies
- **Authentication bypass** — altering session tokens or authorization headers

Example modification: Changed the `User-Agent` header to impersonate a different browser:

```http
User-Agent: Googlebot/2.1 (+http://www.google.com/bot.html)
```

Then clicked **"Forward"** in Burp to send the modified request to the server. The server processed it as if a Google bot had made the request — demonstrating how trivially HTTP headers can be spoofed.

**Step 8 — HTTP history: reviewing all past traffic**

Navigated to **Proxy → HTTP history** — this tab logs every single request/response that has passed through the proxy, even when interception is turned off.

| Column | What It Reveals |
|---|---|
| **#** | Sequential request number |
| **Host** | Target server |
| **Method** | GET, POST, PUT, DELETE, etc. |
| **URL** | The exact path and query parameters requested |
| **Status** | HTTP response code (200 OK, 302 Redirect, 401 Unauthorized, 403 Forbidden, 404 Not Found) |
| **Length** | Response body size in bytes — **critical for identifying anomalies** |
| **MIME Type** | Content type returned (HTML, JSON, image, etc.) |

> **Why "Length" matters:** During brute-force attacks, most failed attempts will return responses of the same size (e.g., a "login failed" page is always 4,521 bytes). A **successful login** typically returns a different response size (e.g., a redirect page at 312 bytes, or a dashboard at 12,847 bytes). Sorting by Length in the HTTP history or Intruder results instantly highlights the anomaly — the one response that is different from all the others is almost certainly the successful attempt.

### Part 3 — Burp Suite Repeater: Manual Request Replaying

**Step 9 — Sending requests to Repeater**

The **Repeater** module allows you to take any intercepted request, modify it, and resend it as many times as needed — without having to re-navigate in the browser. This is essential for:
- Testing how the server responds to different parameter values
- Exploring edge cases (empty fields, special characters, SQL injection payloads)
- Observing how headers like `Cookie` or `Authorization` affect the response

**Workflow:**
1. In **Proxy → HTTP history**, right-clicked on the DVWA login POST request
2. Selected **"Send to Repeater"**
3. Switched to the **Repeater** tab

The request was now loaded and fully editable:

```http
POST /dvwa/login.php HTTP/1.1
Host: 192.0.2.10
Content-Type: application/x-www-form-urlencoded
Content-Length: 44
Cookie: security=low; PHPSESSID=abc123def456

username=admin&password=password&Login=Login
```

**Step 10 — Modifying and replaying**

Changed the `password` parameter to test different credentials manually:

```http
username=admin&password=admin&Login=Login
```

Clicked **"Send"** — the response panel immediately showed the server's reply:

```http
HTTP/1.1 302 Found
Location: index.php
```

A `302 Found` redirect to `index.php` indicated a **successful login** — the server accepted `admin:admin` as valid credentials (DVWA default). In contrast, an incorrect password returned:

```http
HTTP/1.1 200 OK
Content-Length: 4521
```

The `200 OK` with a large body was the login page being re-rendered with a "Login failed" message.

> **Key insight:** The HTTP status code alone is not always sufficient to determine success. In this case, both success (302) and failure (200) returned "successful" HTTP responses. The actual indicator was the **redirect behavior** and the **response body size difference**.

**Step 11 — Rendering the response**

Burp Suite's Repeater includes a **"Render"** tab alongside the raw response. Clicking it renders the HTML response as it would appear in a browser — allowing visual inspection without leaving Burp.

This is particularly useful when:
- The raw HTML is too dense to read quickly
- You need to verify whether an XSS payload was reflected and rendered
- You want to see error messages or form changes visually

### Part 4 — Burp Suite Intruder: Automated Attack Workflows

**Step 12 — Understanding Intruder's purpose**

The **Intruder** module is Burp's automated attack engine. It takes a base request, marks specific parameters as "insertion points" (payload positions), and then iterates through a list of payloads (wordlists, numbers, patterns) — sending one modified request for each payload value.

**Primary use cases:**
- **Brute-force attacks** — cycling through username/password lists
- **Fuzzing** — testing parameters with unexpected inputs (SQL injection, XSS payloads)
- **Enumeration** — iterating through IDs, filenames, or directory names

**Step 13 — Configuring an Intruder attack (login brute-force)**

**Sending the request to Intruder:**
1. From **Proxy → HTTP history**, right-clicked the login POST request
2. Selected **"Send to Intruder"**

**Setting payload positions:**
Navigated to **Intruder → Positions** tab. Burp automatically marks potential insertion points with `§` delimiters. Clicked **"Clear §"** to remove auto-detected positions, then manually selected only the `password` field value:

```http
POST /dvwa/login.php HTTP/1.1
Host: 192.0.2.10
Content-Type: application/x-www-form-urlencoded
Cookie: security=low; PHPSESSID=abc123def456

username=admin&password=§password§&Login=Login
```

The `§password§` marks tell Intruder: "Replace this value with each payload from my list."

**Step 14 — Intruder attack types**

Burp Intruder offers four attack types, each designed for different scenarios:

| Attack Type | How It Works | Use Case |
|---|---|---|
| **Sniper** | Uses a single payload list. Iterates through each position one at a time, keeping other positions at their default values. | Testing one parameter at a time (e.g., brute-forcing just the password field) |
| **Battering Ram** | Uses a single payload list. Inserts the **same payload** into **all positions** simultaneously. | When the same value needs to appear in multiple fields (e.g., testing if username equals password) |
| **Pitchfork** | Uses **one payload list per position**. Iterates through all lists in parallel (payload 1 from list A + payload 1 from list B, then payload 2 from list A + payload 2 from list B, etc.). | Credential stuffing with known username:password pairs |
| **Cluster Bomb** | Uses **one payload list per position**. Tests **every combination** of payloads across all positions (Cartesian product). | Full brute-force of multiple parameters (e.g., all usernames × all passwords) |

For this login brute-force with a known username (`admin`) and an unknown password, **Sniper** was the correct choice — only one position needed to be iterated.

**Step 15 — Loading a wordlist**

Navigated to **Intruder → Payloads** tab:
1. Set **Payload type** to "Simple list"
2. Loaded a wordlist — options include:
   - Manually typing entries one by one
   - Pasting a list from clipboard
   - Loading from file (e.g., `/usr/share/wordlists/rockyou.txt` or a custom smaller list)

For this exercise, used a small custom wordlist to avoid the rate-limiting of Community Edition:

```
password
admin
123456
letmein
test
root
msfadmin
toor
```

**Step 16 — Launching the attack and analyzing results**

Clicked **"Start attack"** — Intruder opened a new results window showing each request, its payload, response status, and response length.

| Request # | Payload | Status | Length |
|---|---|---|---|
| 1 | password | 302 | 312 |
| 2 | admin | 302 | 312 |
| 3 | 123456 | 200 | 4521 |
| 4 | letmein | 200 | 4521 |
| 5 | test | 200 | 4521 |
| 6 | root | 200 | 4521 |
| 7 | msfadmin | 200 | 4521 |
| 8 | toor | 200 | 4521 |

**How to identify the successful payload:**

1. **Sort by Length (ascending or descending):** The successful login returns a `302` redirect with a body of only 312 bytes (the redirect page). All failed attempts return `200 OK` with 4,521 bytes (the full login page with error message). The size anomaly immediately reveals the winner.
2. **Sort by Status:** The `302` status code (redirect to dashboard) stands out among the `200` responses.
3. **Inspect the response:** Clicking on request #1 and viewing the response confirms `Location: index.php` — a redirect to the authenticated area.

> **Result:** Payloads `password` and `admin` both successfully authenticated. This reveals that DVWA has multiple valid credentials — a critical finding in a real assessment.

**Step 17 — Intruder with Cluster Bomb (multi-parameter brute-force)**

To demonstrate Cluster Bomb, configured Intruder with two payload positions:

```http
username=§admin§&password=§password§&Login=Login
```

- **Payload Set 1 (username):** `admin`, `user`, `guest`, `root`
- **Payload Set 2 (password):** `password`, `admin`, `123456`, `test`

Cluster Bomb generates every combination: 4 usernames × 4 passwords = **16 total requests**. This is computationally expensive at scale — a 1,000-word username list × a 10,000-word password list produces 10,000,000 requests.

### Part 5 — Extracting Intelligence from Requests and Responses

**Step 18 — Information leakage in HTTP headers**

By examining the server's response headers in Repeater, identified significant information leakage:

```http
HTTP/1.1 200 OK
Server: Apache/2.2.8 (Ubuntu) DAV/2
X-Powered-By: PHP/5.2.4-2ubuntu5.10
Set-Cookie: PHPSESSID=abc123def456; path=/
```

| Header | Intelligence Gained |
|---|---|
| `Server: Apache/2.2.8 (Ubuntu)` | Exact web server version and OS — enables targeted CVE searching |
| `X-Powered-By: PHP/5.2.4` | Exact PHP version — PHP 5.2.4 has known critical vulnerabilities |
| `Set-Cookie: PHPSESSID=...` | Session management mechanism — can test for session fixation |
| Absence of `X-Frame-Options` | Potentially vulnerable to clickjacking attacks |
| Absence of `Content-Security-Policy` | No CSP — easier to exploit XSS if found |
| Absence of `Strict-Transport-Security` | No HSTS — vulnerable to SSL stripping |

**Step 19 — Using the Decoder module**

Burp's **Decoder** module converts data between different encoding formats. This is useful when parameters are Base64-encoded, URL-encoded, or hex-encoded:

- **Encode:** Convert plaintext → Base64, URL-encoding, hex, HTML entities
- **Decode:** Convert encoded strings back to readable plaintext
- **Hash:** Generate MD5, SHA-1, SHA-256 hashes of any input

Example: Decoded a Base64-encoded cookie value to reveal its plaintext contents — exposing that the session cookie contained the username in cleartext (a severe vulnerability).

**Step 20 — Using the Comparer module**

The **Comparer** module performs a byte-level or word-level diff between two responses. This is invaluable for:
- Comparing a successful vs. failed login response to identify the exact difference
- Detecting subtle changes in response bodies after modifying a single parameter
- Identifying anti-CSRF token changes between requests

## Challenge Encountered

> **Obstacle:** After configuring the browser proxy, all pages loaded extremely slowly, and some external resources (CSS, JS, images) failed to load entirely.

**Root cause:** Burp Suite's **"Intercept is on"** mode was catching every single request — including all CSS files, JavaScript files, images, and favicon requests. Each one had to be manually forwarded before the next could be processed, creating a bottleneck.

---

## Resolution

1. Configured scope rules in **Target → Scope** to limit Burp's interception to only the target (`192.0.2.10`).
2. In **Proxy → Options → Intercept Client Requests**, enabled the rule: "And URL is in target scope" — this tells Burp to only hold requests destined for the defined target and automatically forward everything else.
3. Additionally, added file extension filters to skip intercepting `.css`, `.js`, `.png`, `.jpg`, `.gif`, and `.ico` files — these static assets are rarely relevant to security testing.

> **Key realization:** An untargeted proxy configuration is unusable in practice. Scoping the target and filtering static content is not optional — it is the first thing to configure after setting up the proxy, before any actual testing begins.

---

## Key Takeaways

- **Response size is the primary indicator during brute-force attacks.** HTTP status codes can be misleading (both success and failure may return 200). The response body length is the most reliable differentiator — the one outlier in a sea of identical sizes is almost certainly the hit.
- **Intruder attack types are not interchangeable.** Sniper for single-parameter testing, Pitchfork for correlated pairs, and Cluster Bomb for exhaustive combinations. Choosing the wrong type wastes time or misses the target entirely.
- **Server headers leak critical intelligence.** A single response can reveal the exact web server version, language runtime, OS, and the absence of security headers — providing a complete attack surface map without any active exploitation.
- **Never expose a deliberately vulnerable VM to the network.** Metasploitable 2 is designed to be compromised. Host-Only networking is a non-negotiable lab configuration.
