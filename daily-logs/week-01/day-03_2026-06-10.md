## Technical Log

**Step 1 — Nmap fundamentals: what it does and how it works**

Nmap (Network Mapper) is the industry-standard tool for network discovery and security auditing. At its core, it sends specially crafted packets to target hosts and analyzes the responses to determine:
- Which hosts are alive on a network
- Which ports are open, closed, or filtered
- What services and versions are running on those ports
- What operating system the target is running

```bash
# Basic scan against a single target (sanitized IP)
nmap 192.0.2.1
```

By default, Nmap scans the top 1,000 most common TCP ports. The output categorizes each port into one of six states:

| Port State | Meaning |
|---|---|
| **open** | A service is actively accepting connections on this port |
| **closed** | The port is accessible but no service is listening — responds with RST |
| **filtered** | Nmap cannot determine the state; packets are being dropped, likely by a firewall |
| **unfiltered** | The port is accessible but Nmap cannot determine if it is open or closed (ACK scan only) |
| **open\|filtered** | Nmap cannot distinguish between open and filtered (UDP/IP protocol scans) |
| **closed\|filtered** | Nmap cannot distinguish between closed and filtered (IP ID idle scan only) |

**Step 2 — TCP scan types and how they differ**

This was the most technically dense part of the day. Each scan type manipulates the TCP handshake differently, producing distinct packet patterns. Running Wireshark alongside Nmap made these differences visible at the packet level.

### TCP Connect Scan (`-sT`)

The simplest scan. Nmap completes the full TCP three-way handshake for every port.

```bash
nmap -sT 192.0.2.1
```

Packet behavior observed in Wireshark:
```
Client → Target: SYN
Target → Client: SYN-ACK    (port is OPEN)
Client → Target: ACK         (handshake complete)
Client → Target: RST         (Nmap tears down the connection)
```

If the port is closed:
```
Client → Target: SYN
Target → Client: RST-ACK     (port is CLOSED — immediate rejection)
```

This scan is reliable but noisy — it creates full connections that are logged by the target system. It does not require root privileges because it uses the operating system's standard `connect()` system call.

### SYN Scan (`-sS`) — "Half-Open" or "Stealth" Scan

The default scan when running Nmap with root privileges. It never completes the handshake.

```bash
sudo nmap -sS 192.0.2.1
```

Packet behavior observed in Wireshark:
```
Client → Target: SYN
Target → Client: SYN-ACK    (port is OPEN)
Client → Target: RST         (Nmap resets immediately — no handshake completion)
```

Because the three-way handshake is never completed, many older logging mechanisms that only record established connections will not log this scan. However, modern IDS/IPS systems detect SYN scans easily. The term "stealth scan" is historical and somewhat misleading in contemporary environments.

Requires root/sudo because it constructs raw packets directly rather than using the OS network stack.

### UDP Scan (`-sU`)

Scans UDP ports instead of TCP. UDP is connectionless, so the behavior is fundamentally different.

```bash
sudo nmap -sU 192.0.2.1
```

Packet behavior:
```
Client → Target: UDP packet to port
No response                  → port is open|filtered (ambiguous)
Target → Client: ICMP Port Unreachable → port is CLOSED
```

UDP scanning is inherently slower and less reliable than TCP scanning because the absence of a response is ambiguous — the packet may have been dropped by a firewall, or the service may simply not send a reply. Nmap compensates by retransmitting and by using version detection payloads for known UDP services (DNS on 53, SNMP on 161, DHCP on 67/68).

### ACK Scan (`-sA`)

Does not determine if ports are open. Instead, it maps firewall rulesets — specifically, which ports are filtered versus unfiltered.

```bash
sudo nmap -sA 192.0.2.1
```

Packet behavior:
```
Client → Target: ACK
Target → Client: RST         (port is UNFILTERED — the ACK reached the target)
No response / ICMP error     (port is FILTERED — a firewall dropped the ACK)
```

This is a reconnaissance technique for understanding firewall behavior, not for finding open services.

### FIN, NULL, and Xmas Scans (`-sF`, `-sN`, `-sX`)

These exploit a nuance in RFC 793: compliant TCP implementations should respond to packets with no SYN, RST, or ACK flags by sending RST for closed ports and silently dropping for open ports.

```bash
sudo nmap -sF 192.0.2.1     # FIN scan — sends only FIN flag
sudo nmap -sN 192.0.2.1     # NULL scan — sends no flags at all
sudo nmap -sX 192.0.2.1     # Xmas scan — sets FIN, PSH, URG flags
```

Xmas scan is named because the packet with FIN+PSH+URG set "lights up like a Christmas tree" in a protocol analyzer.

These scans are less reliable on modern systems (Windows ignores RFC 793 for these cases and sends RST regardless), but they can evade some stateless firewalls that only watch for SYN packets.

**Step 3 — Port specification options**

```bash
# Scan specific ports
nmap -p 22,80,443 192.0.2.1

# Scan a port range
nmap -p 1-1024 192.0.2.1

# Scan all 65,535 ports
nmap -p- 192.0.2.1

# Scan top N most common ports
nmap --top-ports 100 192.0.2.1

# Combine TCP and UDP port scanning
nmap -sS -sU -p T:80,443,U:53,161 192.0.2.1
```

The `-p-` flag is important for thorough assessments. The default top-1000 scan misses any service running on a non-standard port. In real-world penetration testing, services are frequently found on unusual ports specifically to avoid default scans.

**Step 4 — Service, Version, and OS Detection**

```bash
# Detect service versions running on open ports
nmap -sV 192.0.2.1

# Aggressive version detection (more probes, slower)
nmap -sV --version-intensity 9 192.0.2.1

# OS detection (TCP/IP Fingerprinting)
sudo nmap -O 192.0.2.1

# Aggressive scan: OS detection + version detection + scripts + traceroute
sudo nmap -A 192.0.2.1
```

**OS Detection (`-O`)**: This option uses TCP/IP stack fingerprinting. Nmap sends a series of specialized TCP and UDP packets to the target and analyzes the responses. Because different operating systems (and versions) implement the TCP/IP stack slightly differently—such as how they set initial Time-To-Live (TTL) values, TCP window sizes, or flag combinations—Nmap can match the responses against a database to guess the OS with high accuracy. This requires root privileges because it relies on raw packet manipulation.

**Aggressive Scan (`-A`)**: This is a macro flag that combines four critical reconnaissance features into a single command:
1. **OS Detection** (`-O`)
2. **Version Detection** (`-sV`)
3. **Script Scanning** (`-sC` using the `default` script category)
4. **Traceroute** (`--traceroute` to map network hops)

While `-A` is highly convenient and provides a comprehensive initial picture of a target, it is extremely noisy and slow. It generates significant network traffic and will almost certainly be flagged by any rudimentary IDS/IPS.

**Step 5 — Nmap Scripting Engine (NSE)**

NSE extends Nmap far beyond a simple port scanner, turning it into a powerful platform for vulnerability detection, network discovery, and even exploitation. It uses scripts written in the Lua programming language.

### Where NSE scripts are located

On Arch Linux, NSE scripts are stored at:

```bash
ls /usr/share/nmap/scripts/ | head -20
# Output: hundreds of .nse files

# Count total scripts available
ls /usr/share/nmap/scripts/*.nse | wc -l
# Output: 600+ scripts
```

Each script is essentially a modular plugin. Nmap manages script execution, providing a standardized API for network I/O, allowing script authors to focus on the vulnerability or protocol logic rather than low-level socket programming.

You can view a script's documentation via `--script-help`:

```bash
nmap --script-help http-title
# Shows: Retrieves the title of the web page from HTTP services
```

### Script categories

Scripts are categorized to allow bulk execution of related tasks. Key categories include:

| Category | Purpose |
|---|---|
| `default` | Safe, generally useful scripts (run with `-sC`) |
| `vuln` | Actively checks for known vulnerabilities (e.g., MS17-010, Shellshock) |
| `exploit` | Attempts to actively exploit a found vulnerability |
| `brute` | Performs brute-force credential testing against services (FTP, SSH, etc.) |
| `discovery` | Extensive querying of public registries, SNMP, or directory services |
| `safe` | Scripts that are strictly passive and guaranteed not to crash the target |

### Running NSE scripts

```bash
# Run the default, safe scripts (equivalent to -sC)
nmap --script=default 192.0.2.1

# Run a specific script
nmap --script=http-title 192.0.2.1

# Run all scripts in the 'vuln' category
nmap --script=vuln 192.0.2.1

# Run multiple specific scripts simultaneously
nmap --script=http-title,http-headers,ssl-cert 192.0.2.1
```

For a concrete example: if port 80 is open, running `nmap --script=http-enum 192.0.2.1` will automatically spider the web server and identify common directories (like `/admin`, `/login`, or `/wp-content`), effectively combining port scanning with basic directory brute-forcing.

### Script file structure

Examined the contents of a script file to understand the format:

```bash
head -30 /usr/share/nmap/scripts/http-title.nse
```

Each `.nse` file contains:
1. A **description** block.
2. A **categories** table assigning it to groups like "default" or "discovery".
3. A **rule** function (a Lua boolean function that determines *if* the script should execute—e.g., "only run if port 80 is open and service is HTTP").
4. An **action** function (the actual logic that connects to the port, sends the payload, and parses the response).

**Step 6 — Combining Options for Efficiency**

The true power of Nmap comes from chaining these options together to build highly specific, efficient scanning workflows. A professional workflow rarely runs Nmap just once; it usually involves a broad discovery scan followed by targeted, in-depth scans.

```bash
# Workflow Example 1: The "Sweep then Deep" approach
# Phase 1: Fast ping sweep to find live hosts on a /24 subnet (no port scanning)
nmap -sn 192.0.2.0/24

# Phase 2: Targeted SYN scan on a discovered host, scanning all ports fast
sudo nmap -sS -p- --min-rate 5000 192.0.2.1

# Phase 3: Deep dive into the specific open ports found (e.g., 22, 80)
# Combines SYN scan, version detection, default scripts, specific ports, and outputting to a file
sudo nmap -sS -sV -sC -p 22,80 -oN targeted_scan.txt 192.0.2.1
```

Key combination modifiers:
- `-oA <filename>`: Output All formats (Text, XML, and Greppable). Crucial for documenting findings without running the noisy scan twice.
- `--min-rate <number>`: Forces Nmap to send a minimum number of packets per second, drastically speeding up scans (at the risk of overwhelming weak networks or missing dropped packets).
- `-T<0-5>`: Timing templates. `-T4` is the standard "aggressive/fast" template used when network latency is low.

**Step 7 — Running Wireshark in parallel**

Opened Wireshark on the active network interface before executing each Nmap scan. This made the theoretical differences between scan types immediately visible.

```bash
# Start Wireshark on the primary interface (requires root or wireshark group membership)
sudo wireshark &
```

Applied display filters to isolate Nmap traffic:

```
# Filter for traffic to/from the target
ip.addr == 192.0.2.1

# Filter for SYN packets only (useful during SYN scan)
tcp.flags.syn == 1 && tcp.flags.ack == 0

# Filter for RST packets (reveals closed ports and half-open teardowns)
tcp.flags.reset == 1

# Filter for ICMP (reveals filtered ports during UDP scan)
icmp
```

Key observation: during a SYN scan of a target with 3 open ports out of 1,000 scanned, Wireshark showed exactly 1,000 outgoing SYN packets, 3 SYN-ACK responses (one per open port), 997 RST-ACK responses (one per closed port), and 3 outgoing RST packets (Nmap tearing down the half-open connections). The packet count matched the theory precisely.

## Challenge Encountered

> **Obstacle:** Running `nmap -sS 192.0.2.1` without `sudo` produced an error: `TCP connect scan requested (despite -sS flag) because you are not root`. Nmap silently fell back to a TCP Connect scan.

**Root cause:** SYN scanning requires raw socket access to craft custom TCP packets with only the SYN flag set. On Linux, raw socket creation requires either root privileges or the `CAP_NET_RAW` capability. Without it, Nmap cannot construct the packets directly and falls back to using the operating system's `connect()` call, which performs a full three-way handshake (TCP Connect scan).

---

## Resolution

1. Ran the scan with `sudo nmap -sS 192.0.2.1` — the scan executed correctly as a SYN scan.
2. Verified in Wireshark that the packets now showed the half-open pattern (SYN → SYN-ACK → RST) instead of the full handshake.
3. As an alternative to `sudo`, Nmap can be granted raw socket capability permanently: `sudo setcap cap_net_raw+ep /usr/bin/nmap`. This was noted but not applied, as it has security implications of its own.

> **Key realization:** The privilege model in Linux directly affects what tools can do at the network level. Understanding why a tool needs root access — not just that it does — is essential for making informed decisions about privilege management.

---

## Key Takeaways

- **SYN scan is default for a reason.** It is faster than Connect scan (no handshake overhead), less likely to be logged by the target's application-layer logging, and provides the same port state information. It requires root, but that is the correct tradeoff for a scanning tool.
- **UDP scanning is fundamentally unreliable.** The absence of a response is ambiguous — it could mean open, filtered, or simply dropped. Combine `-sU` with `-sV` to send protocol-specific payloads that increase accuracy.
- **NSE turns Nmap from a scanner into a platform.** The 600+ built-in scripts handle tasks from banner grabbing to vulnerability detection. Learning to use `--script=vuln` and `--script-help` effectively saves significant time during assessments.
