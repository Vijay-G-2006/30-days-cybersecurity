## Technical Log

**Step 1 — Understanding why custom scripts matter**

Day 5 introduced Hashcat — a powerful, GPU-accelerated cracking tool. However, relying solely on pre-built tools creates a dependency without understanding the underlying process. Writing a custom script forces an understanding of:
- How hash functions accept input (bytes, not strings)
- How the comparison logic works (string matching of hex digests)
- Why weak inputs are vulnerable (small keyspace = fast iteration)

**Step 2 — Python's hashlib module: generating hashes programmatically**

The `hashlib` module is part of Python's standard library — no installation required. It provides a unified interface to multiple hash algorithms.

```python
import hashlib

# Generate a SHA-512 hash of the string "12345"
input_string = "12345"
input_bytes = input_string.encode('utf-8')  # hashlib requires bytes, not str
hash_object = hashlib.sha512(input_bytes)
hex_digest = hash_object.hexdigest()

print(hex_digest)
# Output: 3627909a29c31381a071ec27f7c9ca97726182aed29a7dbe9a35dc6...
```

Key details:
- **`.encode('utf-8')`** — Hash functions operate on raw bytes. Forgetting to encode the string raises a `TypeError`. This is the Python equivalent of the `-n` flag lesson from Day 5 (`echo -n`).
- **`.hexdigest()`** — Returns the hash as a human-readable hexadecimal string (the format used for comparison and storage).
- **`.digest()`** — Returns the hash as raw bytes (used in cryptographic protocols, not for display).

Other supported algorithms:

```python
# Available algorithms on this system
print(hashlib.algorithms_available)
# {'sha512', 'sha256', 'sha1', 'md5', 'sha384', 'sha224', 'blake2b', 'blake2s', ...}

# Generate hashes with different algorithms
hashlib.md5(b"hello").hexdigest()       # MD5 (32 hex chars)
hashlib.sha256(b"hello").hexdigest()    # SHA-256 (64 hex chars)
hashlib.sha512(b"hello").hexdigest()    # SHA-512 (128 hex chars)
```

**Step 3 — Building the brute-force script**

The core logic of any hash cracker is a loop: generate a candidate, hash it, compare it to the target. If they match, the original input has been found.

The scenario: given a SHA-512 hash that is known to have been generated from a number between 1 and 9,999,999, find the original number.

```python
import hashlib
import time

def brute_force_numeric_hash(target_hash: str, max_range: int = 9999999) -> None:
    """
    Demonstrates how weak passwords (like short numbers)
    can be brute-forced by hashing predictable inputs and comparing them.
    """
    print(f"[*] Target hash: {target_hash[:20]}...")
    print(f"[*] Searching range: 1 to {max_range:,}")
    start_time = time.time()

    for number in range(1, max_range + 1):
        # 1. Convert number to string, then encode to bytes
        input_bytes = str(number).encode('utf-8')

        # 2. Generate SHA-512 hash and get hex representation
        resultant_hash = hashlib.sha512(input_bytes).hexdigest()

        # 3. Compare with target
        if resultant_hash == target_hash:
            elapsed_time = time.time() - start_time
            print(f"\n[+] Match found!")
            print(f"[+] Original input: {number}")
            print(f"[+] Time elapsed: {elapsed_time:.2f} seconds")
            print(f"[+] Attempts: {number:,}")
            return

    elapsed_time = time.time() - start_time
    print(f"\n[-] No match found in range 1–{max_range:,}.")
    print(f"[-] Time elapsed: {elapsed_time:.2f} seconds")


if __name__ == "__main__":
    # SHA-512 hash of a numeric input (example for testing)
    example_hash = (
        "18518656fba871522838d32029e9374606125bb637f13a7547"
        "59c2200af8e76e5e1be01db7f7c"
    )

    brute_force_numeric_hash(example_hash)
```

**Step 4 — Executing and analyzing performance**

```bash
# Run the script
python3 hash_cracker.py
```

The script iterates through numbers sequentially, hashing each one with SHA-512 and comparing it to the target. On a modern CPU, Python can compute approximately 200,000–500,000 SHA-512 hashes per second. This means:
- A 6-digit number (max 999,999) is cracked in **under 5 seconds**
- A 7-digit number (max 9,999,999) is cracked in **under 50 seconds**

This demonstrates exactly why numeric PINs and short passwords are catastrophically weak — even a basic Python script (not optimized, not GPU-accelerated) can exhaust the entire keyspace in seconds.

**Step 5 — Comparing the approach: Python hashlib vs. Hashcat**

| Aspect | Python hashlib Script | Hashcat |
|---|---|---|
| **Speed** | ~200K–500K hashes/sec (CPU only) | Billions of hashes/sec (GPU-accelerated) |
| **Flexibility** | Fully customizable logic, any algorithm, any input pattern | Predefined attack modes with mask/rule syntax |
| **Use case** | Learning, prototyping, custom workflows | Production-grade password recovery |
| **Dependencies** | None (stdlib) | Requires compatible GPU drivers (OpenCL/CUDA) |

> **Key insight:** Hashcat is the right tool for real cracking. Python hashlib is the right tool for understanding *why* Hashcat works. Both are essential — one for speed, the other for comprehension.

**Step 6 — Extending to other hash algorithms**

Modified the script to accept the algorithm as a parameter, making it reusable across MD5, SHA-256, and SHA-512:

```python
import hashlib

def crack_hash(target_hash: str, algorithm: str = "sha512", max_range: int = 9999999):
    """Brute-force a numeric hash using a specified algorithm."""
    hash_func = getattr(hashlib, algorithm)

    for number in range(1, max_range + 1):
        candidate_hash = hash_func(str(number).encode('utf-8')).hexdigest()
        if candidate_hash == target_hash:
            return number
    return None
```

This pattern — using `getattr(hashlib, algorithm)` — dynamically selects the hash function at runtime, avoiding separate functions for each algorithm.

## Challenge Encountered

> **Obstacle:** The initial version of the script printed every single attempt (`[-] trying: 1`, `[-] trying: 2`, ...) to the terminal. When running against a range of 9,999,999 numbers, the script ran extremely slowly — taking over 10 minutes instead of the expected sub-minute execution.

**Root cause:** Terminal I/O is orders of magnitude slower than computation. Printing a line for each of the 9,999,999 iterations created a massive bottleneck. The hash computation itself was fast; the `print()` call for each iteration was the actual performance killer.

---

## Resolution

1. Removed the per-attempt `print()` statement from the loop entirely.
2. The script now runs silently during iteration and only prints output when a match is found or the range is exhausted.
3. Added timing with `time.time()` to measure actual execution performance.

> **Key realization:** I/O operations (printing, writing to files, network calls) inside tight loops are a common performance anti-pattern. In security tooling especially, where millions of iterations are standard, keeping the hot loop free of I/O is critical. This is the same reason Hashcat runs silently and only reports results.

---

## Key Takeaways

- **Numeric passwords are trivially crackable.** Even without GPU acceleration, a basic Python script can exhaust a 7-digit numeric keyspace in under a minute. PINs and short numeric passwords provide effectively zero protection against automated attacks.
- **Hash functions require bytes, not strings.** The `.encode('utf-8')` step is not optional — it is fundamental to how cryptographic functions process data. Forgetting this is the Python equivalent of the `echo -n` pitfall from Day 5.
- **I/O inside hot loops destroys performance.** The difference between printing every iteration and printing only results was a 10x+ speed improvement. Efficient tooling minimizes unnecessary output during computation-heavy operations.
