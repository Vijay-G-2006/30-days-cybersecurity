## Technical Log

**Step 1 — Linux filesystem navigation and file operations**

Started with the fundamentals of moving through the Linux filesystem. The key commands that formed the basis of all subsequent work:

```bash
# Directory navigation
pwd                         # Print current working directory
ls -la                      # List all files including hidden, with permissions
cd /etc                     # Change to a specific directory
cd ~                        # Return to home directory

# File operations
touch testfile.txt          # Create an empty file
mkdir -p projects/security  # Create nested directories
cp file.txt backup/         # Copy files
mv old.txt new.txt          # Rename or move files
rm -rf temp/                # Remove directories recursively
```

The `-la` flag combination on `ls` was immediately useful — it revealed hidden dotfiles (`.bashrc`, `.config/`) that are invisible with a plain `ls`. Understanding that Linux treats everything as a file, including devices and processes, was a foundational realization.

**Step 2 — File permissions and ownership**

Examined how Linux controls access through the permission model:

```bash
# View permissions
ls -l document.txt
# Output: -rw-r--r-- 1 user group 4096 Jun 08 09:30 document.txt

# Modify permissions
chmod 755 script.sh         # rwxr-xr-x — owner full, group/others read+execute
chmod u+x script.sh         # Add execute permission for owner only

# Change ownership
chown user:group file.txt   # Change owner and group
```

The three permission groups (owner, group, others) and three permission types (read=4, write=2, execute=1) use an octal notation system. `755` means the owner can read, write, and execute, while group and others can only read and execute. This directly relates to security — misconfigured permissions are a common attack vector.

**Step 3 — Process and system inspection**

```bash
# Process management
ps aux                      # List all running processes
top                         # Real-time process monitor
kill -9 <PID>               # Force terminate a process
systemctl status sshd       # Check service status (systemd)

# System information
uname -a                    # Kernel and OS information
df -h                       # Disk usage in human-readable format
free -h                     # Memory usage
```

**Step 4 — Networking commands on Arch Linux**

Arch Linux uses `iproute2` as the default networking toolkit. The older `ifconfig` and `netstat` commands from `net-tools` are deprecated in favor of `ip` and `ss`:

```bash
# View network interfaces and IP addresses
ip addr show                # All interfaces with assigned IPs
ip link show                # Interface status (UP/DOWN)

# View routing table
ip route show               # Active routes, including default gateway

# Check active connections
ss -tuln                    # TCP/UDP listening ports
# -t: TCP, -u: UDP, -l: listening, -n: numeric (no DNS resolution)

# Test network connectivity
ping -c 4 192.0.2.1         # Send 4 ICMP echo requests
curl -I https://example.com # Fetch HTTP headers only
```

The `ss -tuln` command was particularly informative — it showed every service listening on the machine and on which port. This is a critical command for security auditing: any unexpected listening port is a potential exposure.

**Step 5 — TCP/IP model: understanding how data travels**

The TCP/IP model is the practical networking model that the internet runs on. It has four layers, each with a distinct responsibility:

| Layer | Name | Function | Key Protocols |
|---|---|---|---|
| 4 | Application | User-facing data exchange | HTTP, HTTPS, DNS, SSH, FTP |
| 3 | Transport | End-to-end delivery, reliability | TCP, UDP |
| 2 | Internet | Logical addressing, routing | IP (IPv4/IPv6), ICMP |
| 1 | Network Access | Physical transmission | Ethernet, Wi-Fi, ARP |

The distinction between TCP and UDP at the Transport layer is critical for security:
- **TCP** (Transmission Control Protocol) — Connection-oriented. Uses a three-way handshake (SYN → SYN-ACK → ACK) to establish a reliable connection. Guarantees delivery and ordering. Used by HTTP, SSH, FTP.
- **UDP** (User Datagram Protocol) — Connectionless. No handshake, no delivery guarantee. Faster but unreliable. Used by DNS, DHCP, video streaming.

From a security perspective, the TCP three-way handshake is relevant because SYN flood attacks exploit this mechanism by sending massive numbers of SYN packets without completing the handshake, exhausting server resources.

**Step 6 — IP addresses: structure and classification**

An IPv4 address is a 32-bit number divided into four octets (e.g., `192.168.1.10`). Each octet ranges from 0 to 255.

**IP address classes:**

| Class | Range | Default Subnet Mask | Purpose |
|---|---|---|---|
| A | 1.0.0.0 – 126.255.255.255 | 255.0.0.0 (/8) | Large networks |
| B | 128.0.0.0 – 191.255.255.255 | 255.255.0.0 (/16) | Medium networks |
| C | 192.0.0.0 – 223.255.255.255 | 255.255.255.0 (/24) | Small networks |
| D | 224.0.0.0 – 239.255.255.255 | — | Multicast |
| E | 240.0.0.0 – 255.255.255.255 | — | Reserved/experimental |

**Private IP ranges** (RFC 1918) — not routable on the public internet:

| Class | Private Range |
|---|---|
| A | 10.0.0.0 – 10.255.255.255 |
| B | 172.16.0.0 – 172.31.255.255 |
| C | 192.168.0.0 – 192.168.255.255 |

The distinction between public and private IPs is fundamental to network security. Private addresses require NAT (Network Address Translation) to communicate with the internet, which provides a layer of obscurity but is not a substitute for proper firewall rules.

The loopback address `127.0.0.1` refers to the local machine itself and never leaves the network interface. It is commonly used for testing local services.

## Challenge Encountered

> **Obstacle:** Attempted to use `ifconfig` to view network interfaces and received `command not found`.

**Root cause:** Arch Linux does not include the `net-tools` package by default. The `ifconfig` and `netstat` commands are part of the deprecated `net-tools` suite. Arch ships with `iproute2` as the modern replacement.

---

## Resolution

1. Searched the Arch Wiki for the equivalent modern commands.
2. Confirmed that `ip addr show` replaces `ifconfig` and `ss` replaces `netstat`.
3. Verified functionality — `ip addr show` provided more detailed output than `ifconfig` would have, including interface state flags and scope information.

> **Key realization:** Relying on outdated commands is a practical risk. Tools evolve, and the modern equivalents (`ip`, `ss`) are not just replacements — they provide richer output and better scripting support.

---

## Key Takeaways

- **Linux permissions are security-critical:** A file with `777` permissions is world-readable, writable, and executable — effectively an open door. Default to least privilege.
- **TCP vs. UDP is a security decision:** TCP's handshake creates overhead but enables stateful inspection by firewalls. UDP's lack of state makes it harder to filter and easier to spoof.
- **Private IPs are not a security control:** NAT hides internal addresses but does not replace firewalls, access control lists, or proper network segmentation.
