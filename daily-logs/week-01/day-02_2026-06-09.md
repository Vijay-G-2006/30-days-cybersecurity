# Day 2 — Linux Filesystem Hierarchy and Directory Structure

> *Understood why every directory in Linux exists where it does, what it contains, and why the separation matters for system stability and security.*

## Technical Log

**Step 1 — The root of everything: `/`**

Every path in Linux begins at the root directory `/`. Unlike Windows, which uses drive letters (C:\, D:\), Linux presents the entire system as a single unified tree rooted at `/`. Every device, partition, and remote filesystem is mounted somewhere under this tree.

```bash
# View the top-level directory structure
ls -l /
```

This produced a list of approximately 18-20 directories, each with a specific, well-defined role governed by the Filesystem Hierarchy Standard (FHS).

**Step 2 — Mapping every directory and its purpose**

Walked through each top-level directory systematically. Used `ls`, `file`, and `stat` to inspect contents and understand what types of files each directory holds.

### Essential System Directories

**`/bin` — Essential User Binaries**
Contains fundamental command-line utilities required for single-user mode and basic system operation. These are the commands available even when no other filesystems are mounted.

```bash
ls /bin | head -20
# Output includes: bash, cat, cp, ls, mv, rm, chmod, chown, grep, find, mount...
file /bin/ls
# Output: ELF 64-bit LSB pie executable, x86-64 — a compiled binary
```

On Arch Linux, `/bin` is a symlink to `/usr/bin`. This is part of the UsrMerge — a modern consolidation where `/bin`, `/sbin`, and `/lib` all point into `/usr/`. The historical separation existed because `/usr` could be on a separate partition that was not available during early boot.

**`/sbin` — System Administration Binaries**
Contains binaries intended for system administration tasks, typically requiring root privileges.

```bash
ls /sbin | head -10
# Output includes: fdisk, fsck, iptables, mkfs, reboot, shutdown, ip, modprobe
```

The distinction between `/bin` and `/sbin` is one of privilege: `/bin` holds commands any user can run, while `/sbin` holds commands that generally require `sudo` or root access. On Arch, `/sbin` is also symlinked to `/usr/bin`.

**`/boot` — Boot Loader Files**
Contains everything needed to start the system before the kernel hands off control: the Linux kernel image, initial RAM disk (initramfs), and bootloader configuration.

```bash
ls /boot
# Typical contents: vmlinuz-linux, initramfs-linux.img, grub/
```

This directory is security-sensitive. An attacker with write access to `/boot` could replace the kernel or modify GRUB configuration to boot a compromised image. It should be owned by root with restrictive permissions.

### Configuration and Variable Data

**`/etc` — System Configuration Files**
The central location for all system-wide configuration files. The name historically derives from "et cetera" but functionally it means "Editable Text Configuration."

```bash
ls /etc | head -20
# Output includes: passwd, shadow, fstab, hostname, hosts, pacman.conf, ssh/, sudoers
```

Key files examined:

| File | Purpose |
|---|---|
| `/etc/passwd` | User account information (username, UID, GID, home directory, shell) |
| `/etc/shadow` | Hashed passwords — readable only by root |
| `/etc/fstab` | Filesystem mount table — defines what gets mounted at boot |
| `/etc/hosts` | Static hostname-to-IP mappings — checked before DNS |
| `/etc/hostname` | The system's hostname |
| `/etc/sudoers` | Sudo privilege configuration — who can run what as root |
| `/etc/ssh/sshd_config` | SSH daemon configuration |
| `/etc/pacman.conf` | Arch Linux package manager configuration |

```bash
# View user accounts
cat /etc/passwd
# Format: username:x:UID:GID:comment:home_directory:shell

# Verify shadow file is restricted
stat /etc/shadow
# Permissions: -rw------- (readable only by root)
```

From a security perspective, `/etc` is one of the most critical directories. Misconfigured files here — especially `sudoers`, `sshd_config`, or `passwd` — are common vectors for privilege escalation.

**`/var` — Variable Data**
Contains data that changes frequently during system operation: logs, caches, mail spools, and temporary processing files.

```bash
ls /var
# Output includes: log/, cache/, lib/, tmp/, spool/, run/

# View system logs
ls /var/log
# Output includes: pacman.log, journal/, auth.log, syslog
```

Key subdirectories:

| Directory | Contents |
|---|---|
| `/var/log` | System and application log files — the first place to check during incident response |
| `/var/cache` | Application cache data (e.g., downloaded packages in `/var/cache/pacman/pkg/`) |
| `/var/lib` | Persistent application state data (databases, package manager state) |
| `/var/tmp` | Temporary files that should persist across reboots (unlike `/tmp`) |

**`/tmp` — Temporary Files**
A world-writable directory for temporary file storage. Files here are typically cleared on reboot.

```bash
stat /tmp
# Permissions: drwxrwxrwt — note the sticky bit (t)
```

The sticky bit (`t` in the permissions) is a critical security feature: it prevents users from deleting files they do not own, even though the directory is world-writable. Without the sticky bit, any user could delete another user's temporary files.

### User and Application Data

**`/home` — User Home Directories**
Each user account gets a personal directory under `/home` (e.g., `/home/vijay`). Contains user-specific configuration (dotfiles), documents, and personal data.

```bash
ls -la ~/
# Shows dotfiles: .bashrc, .config/, .ssh/, .local/
```

From a security standpoint, `/home` directories should have `700` or `750` permissions. A directory with `755` would allow other users on the system to read its contents.

**`/root` — Root User's Home**
The home directory for the root (superuser) account. Located at `/root` instead of `/home/root` because `/home` might be on a separate partition that is unavailable during recovery operations.

**`/opt` — Optional/Third-Party Software**
Used for self-contained third-party application installations that do not follow the standard `/usr` directory layout. Software installed here keeps all its files in one subdirectory (e.g., `/opt/application_name/`).

**`/srv` — Service Data**
Contains data served by the system — for example, files served by a web server (`/srv/http/`) or FTP server (`/srv/ftp/`). Often empty on workstations.

### Virtual and Special Filesystems

**`/dev` — Device Files**
Contains device nodes — special files that represent hardware devices and virtual interfaces. Linux treats devices as files, which means standard file operations (read, write) can interact with hardware.

```bash
ls /dev | head -15
# Output includes: sda, sda1, sda2, tty0, null, zero, random, urandom

file /dev/null
# Output: character special — not a regular file, it is a device node

# /dev/null discards all data written to it
echo "test" > /dev/null    # Data disappears

# /dev/urandom provides cryptographically secure random bytes
head -c 16 /dev/urandom | xxd
```

Key device files:

| Device | Purpose |
|---|---|
| `/dev/null` | Discards all input — used to suppress output |
| `/dev/zero` | Produces an infinite stream of zero bytes — used for disk wiping |
| `/dev/urandom` | Produces cryptographically secure random data |
| `/dev/sda` | First SATA/SCSI disk (entire device) |
| `/dev/sda1` | First partition on the first disk |
| `/dev/tty` | Current terminal device |

**`/proc` — Process Information (Virtual)**
A virtual filesystem generated by the kernel at runtime. It does not exist on disk. Every running process gets a numbered directory, and system information is exposed as readable files.

```bash
ls /proc | head -20
# Output: numbered directories (one per PID), plus pseudo-files like cpuinfo, meminfo

cat /proc/cpuinfo | head -10  # CPU details
cat /proc/meminfo | head -5   # Memory statistics
cat /proc/version              # Kernel version

# Inspect a specific process
ls /proc/1/                    # PID 1 is systemd (init)
cat /proc/1/cmdline            # Command that started the process
```

`/proc` is valuable for security monitoring. For instance, an attacker running a hidden process still gets a `/proc/<PID>/` entry that reveals its command line, environment variables, and open file descriptors.

**`/sys` — System and Kernel Parameters (Virtual)**
Another virtual filesystem that exposes kernel subsystem information — hardware devices, drivers, and kernel configuration. Used primarily for hardware interaction and kernel tuning.

```bash
ls /sys/class/net/             # List network interfaces
cat /sys/class/net/enp0s3/address  # MAC address of an interface
```

### Libraries and Shared Resources

**`/lib` — Essential Shared Libraries**
Contains shared libraries (`.so` files) required by binaries in `/bin` and `/sbin`. On Arch, symlinked to `/usr/lib` as part of UsrMerge.

```bash
ls /lib | grep libc
# Output: libc.so.6 — the GNU C Library, required by virtually every binary
```

**`/usr` — User System Resources**
The largest directory on most systems. Contains the bulk of installed software, libraries, documentation, and shared resources. It is read-only in normal operation.

| Subdirectory | Contents |
|---|---|
| `/usr/bin` | Installed application binaries |
| `/usr/lib` | Shared libraries for installed applications |
| `/usr/share` | Architecture-independent data (man pages, icons, documentation) |
| `/usr/local` | Locally compiled software (not managed by the package manager) |
| `/usr/include` | C/C++ header files for development |

### Mount Points

**`/mnt` — Temporary Mount Point**
Used for manually mounting filesystems temporarily (e.g., mounting a USB drive or a network share for one-time access).

**`/media` — Removable Media**
Used by desktop environments for auto-mounting removable devices (USB drives, optical discs). On Arch with a minimal setup, this directory is often empty since auto-mounting depends on a desktop environment or `udisks2`.

```bash
# Manual mount example
sudo mount /dev/sdb1 /mnt
ls /mnt
sudo umount /mnt
```

**Step 3 — Understanding why the separation exists**

The directory structure is not arbitrary. It solves three specific engineering problems:

1. **Boot reliability.** `/bin`, `/sbin`, `/etc`, and `/lib` contain everything needed to boot and recover the system. These can exist on the root partition alone. `/usr`, `/home`, and `/var` can be on separate partitions or network mounts — the system can still boot without them.

2. **Permission boundaries.** Configuration (`/etc`) is separated from executables (`/bin`) and user data (`/home`). This allows granular permission policies: users can write to `/home` and `/tmp` but not to `/etc` or `/bin`.

3. **Maintenance isolation.** System packages install into `/usr`. Locally compiled software goes into `/usr/local`. User data lives in `/home`. This means the system can be reinstalled or upgraded without touching user data, and locally compiled software does not conflict with package-managed software.

## Challenge Encountered

> **Obstacle:** Attempted to use `tree /` to visualize the full filesystem hierarchy and the terminal output was overwhelming — thousands of lines of output scrolling indefinitely.

**Root cause:** `tree` recursively lists every file and directory from the given starting point. Running it from `/` means it traverses the entire filesystem, including virtual filesystems like `/proc` and `/sys` that contain thousands of dynamically generated entries.

---

## Resolution

1. Used `tree -L 1 /` to limit output to only the first level of directories.
2. For deeper inspection of specific directories, used `tree -L 2 /etc` to view two levels within a targeted path.
3. Added `--dirsfirst` to sort directories before files for cleaner output.

```bash
tree -L 1 --dirsfirst /     # Top-level overview
tree -L 2 --dirsfirst /etc  # Two levels into /etc
```

> **Key realization:** Scoping tools to the exact level of detail needed is a fundamental habit. Unscoped commands waste time and can obscure the information being sought.

---

## Key Takeaways

- **The filesystem structure is a security architecture.** The separation of configuration (`/etc`), executables (`/bin`), and user data (`/home`) enables granular access control. Merging these would make permission management significantly harder.
- **Virtual filesystems are live system mirrors.** `/proc` and `/sys` are not stored on disk — they are generated by the kernel in real time. They are essential tools for process inspection and hardware enumeration during security assessments.
- **The sticky bit on `/tmp` prevents a class of attacks.** Without it, any user could delete or overwrite temporary files created by other users or services, enabling denial-of-service or race condition exploits.
