## Technical Log

**Step 1 — Adding and configuring users**

Linux user management requires root privileges. The standard utility for creating users is `useradd`.

```bash
# Add a user with a home directory (-m)
sudo useradd -m testuser

# Add a service account without a home directory (-M)
sudo useradd -M svc_account

# Set the default shell for an existing user
sudo usermod -s /bin/bash testuser
```

The distinction between `-m` and `-M` is critical. Interactive human users need a home directory (`/home/username`) to store configuration dotfiles (`.bashrc`, etc.). System service accounts (like a web server user) should never have a home directory or a valid login shell, minimizing the attack surface if the service is compromised.

**Step 2 — Group management and assignment**

Groups simplify permission management by allowing administrators to apply rules to multiple users simultaneously.

```bash
# Create a new group
sudo groupadd developers

# Add a user to an existing group (append with -aG)
sudo usermod -aG developers testuser

# Delete a group
sudo groupdel developers
```

The `-aG` (append to secondary group) flag is a common trap. Running `usermod -G developers testuser` without the `-a` (append) flag will remove the user from all other secondary groups they belong to, which can easily break `sudo` access if they are removed from the `wheel` or `sudo` group.

**Step 3 — Inspecting user and group databases**

Linux does not use a proprietary database for user identities; it uses plain text files located in `/etc`.

```bash
# View the user database
cat /etc/passwd

# View the group database
cat /etc/group
```

Since `/etc/passwd` can be large, I used `head` and `tail` to view specific portions without overwhelming the terminal:

```bash
# View the first 5 lines (usually root and system accounts)
head -n 5 /etc/passwd

# View the last 5 lines (usually newly created users)
tail -n 5 /etc/passwd
```

**Understanding UIDs and GIDs:**
In `/etc/passwd`, each line follows the format: `username:x:UID:GID:comment:home_directory:shell`.
- **UID 0:** Reserved strictly for `root`.
- **UID 1-999:** Reserved for system and service accounts (e.g., `www-data`, `nobody`).
- **UID 1000+:** Assigned to regular, human users. When inspecting a system for unauthorized accounts, looking for UIDs greater than 1000 or unexpected UID 0 accounts is a primary step.

**Step 4 — Switching user contexts**

To test permissions, switching into the newly created user account is necessary.

```bash
# Switch to another user (requires sudo/root to bypass password prompt)
sudo su testuser

# Switch back to the previous user or root
exit
```

**Step 5 — Deleting users securely**

When a user is removed, leaving their home directory behind can lead to orphaned files or data leakage.

```bash
# Delete a user AND remove their home directory and mail spool (-r)
sudo userdel -r testuser
```

**Step 6 — Automating identity provisioning with Bash**

To solidify the concepts, I wrote a bash script to automate user and group creation. The script uses a simple menu to add/delete users and groups, ensuring new users automatically get `/bin/bash` as their shell.

1. Created a file named `provision.sh` with the following logic:

```bash
#!/bin/bash

# Ensure the script is run as root
if [ "$EUID" -ne 0 ]; then
  echo "Please run as root"
  exit 1
fi

# Basic menu interface
echo "1. Add User (with /bin/bash shell)"
echo "2. Delete User (and home directory)"
echo "3. Add Group"
echo "4. Delete Group"
read -p "Select an option (1-4): " option

case $option in
  1)
    read -p "Enter username to add: " username
    useradd -m -s /bin/bash "$username"
    echo "User $username added successfully."
    ;;
  2)
    read -p "Enter username to delete: " username
    userdel -r "$username"
    echo "User $username deleted."
    ;;
  3)
    read -p "Enter group name to add: " groupname
    groupadd "$groupname"
    echo "Group $groupname created."
    ;;
  4)
    read -p "Enter group name to delete: " groupname
    groupdel "$groupname"
    echo "Group $groupname deleted."
    ;;
  *)
    echo "Invalid option."
    ;;
esac
```

2. Made the script executable: `chmod +x provision.sh`.

To turn this local script into a global system command:
```bash
# Remove the .sh extension for a cleaner command name
mv provision.sh provision

# Move it to a directory in the system's PATH
sudo mv provision /usr/local/bin/
```
Now, typing `provision` anywhere in the terminal executes the script. `/usr/local/bin` is the correct location for locally compiled or custom administrative scripts, keeping them separate from package-managed binaries in `/usr/bin`.

**Step 7 — Bash efficiency trick: `sudo !!`**

Encountered the common scenario of typing a command that requires root privileges, but forgetting `sudo`.

```bash
useradd targetuser
# Output: Permission denied

sudo !!
# Executes: sudo useradd targetuser
```
The `!!` history expansion operator recalls the last executed command, making `sudo !!` a highly efficient workflow shortcut.

## Challenge Encountered

> **Obstacle:** After creating a user without specifying a shell, logging into the account resulted in a minimal, featureless prompt (`$`) with no auto-completion or command history.

**Root cause:** On some systems, `useradd` defaults to `/bin/sh` (which is often a symlink to `dash`, a minimal POSIX-compliant shell) rather than `/bin/bash`. 

---

## Resolution

1. Investigated the user's entry in `/etc/passwd` and confirmed the shell was `/bin/sh`.
2. Updated the shell using `sudo usermod -s /bin/bash <username>`.
3. Verified the change by switching users again (`sudo su <username>`), confirming the expected bash prompt and features were active.

> **Key realization:** Never assume defaults. Always explicitly define the shell when creating interactive users, or configure the system's `useradd` defaults file (`/etc/default/useradd`) to specify `/bin/bash`.

---

## Key Takeaways

- **Linux Identity is purely numerical.** The system cares about UIDs and GIDs, not usernames. A user with UID 0 is root, regardless of whether the username is "root" or "support".
- **The Principle of Least Privilege applies to home directories.** Service accounts (`-M`) should never have a home directory or a valid shell.
- **Automation is standard practice.** Moving custom scripts to `/usr/local/bin` demonstrates how easily the Linux environment can be extended and customized to reduce repetitive administrative overhead.
