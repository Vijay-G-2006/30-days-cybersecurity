## Technical Log

**Step 1 — Understanding cryptographic hashing**

Unlike encryption, which is a two-way process designed to be reversed with a key, **hashing is a one-way mathematical function**. It takes an input of any size and produces a fixed-size output (the hash or digest). 

Core properties of a secure hash function:
1. **Deterministic:** The same input will *always* produce the exact same output.
2. **One-way function:** It should be computationally infeasible to reverse the hash back to the original input.
3. **Avalanche effect:** A tiny change in the input (even a single bit) must result in a completely different hash output.
4. **Collision resistance:** It should be exceedingly difficult to find two different inputs that produce the same hash.

**Primary use cases:**
- **Password storage:** Systems store the hash of a password, not the plaintext. During login, the entered password is hashed and compared to the stored hash.
- **File integrity:** Checksums (like SHA-256 hashes of downloaded binaries) verify that a file has not been corrupted or tampered with.
- **Digital signatures:** Used in public key infrastructure (PKI) to verify the authenticity of a message or document.

**Step 2 — Generating hashes from the command line**

Tested hash generation using built-in Linux utilities.

```bash
# Generate an MD5 hash of the string "hello"
echo -n "hello" | md5sum
# Output: 5d41402abc4b2a76b9719d911017c592  -
```

> **Critical Detail:** The `-n` flag in `echo -n` is essential. Without it, `echo` appends a newline character (`\n`) to the end of the string. The hash of `"hello"` is completely different from the hash of `"hello\n"`. Failing to use `-n` is a common pitfall that leads to unmatchable hashes.

**Step 3 — Identifying unknown hashes**

In penetration testing, you rarely know the hash algorithm immediately upon discovering a hash in a database dump. I used two tools to analyze the structure, length, and character set of the "hello" MD5 hash to determine its type.

```bash
# Using hashid with the -m flag (extremely useful for Hashcat)
hashid -m 5d41402abc4b2a76b9719d911017c592
# Output identifies it primarily as MD4/MD5 and provides the Hashcat Mode: [Hashcat Mode: 0]

# Using hash-identifier
hash-identifier
# (Interactive prompt accepts the hash and returns likely algorithms)
```

**Dealing with Discrepancies:**
Because many hash algorithms output exactly the same length and character set (e.g., both MD5 and MD4 produce 32-character hexadecimal strings), `hashid` and `hash-identifier` will sometimes give different top results, or a long list of possibilities. 

When tools disagree, you must deduce the correct type using:
1. **Context:** Was the hash pulled from a Linux `/etc/shadow` file? It's likely SHA-512 (`$6$`). Was it from an old Windows dump? Likely NTLM.
2. **Hashcat Example Hashes:** The most reliable method is cross-referencing the hash format against the official [Hashcat Example Hashes](https://hashcat.net/wiki/doku.php?id=example_hashes) page, which shows exactly what every supported hash looks like.

**Step 4 — Introduction to Hashcat**

Hashcat is the world's fastest password recovery utility, leveraging CPU and GPU processing to crack hashes. Its operation relies heavily on two primary flags:
- `-m` (Hash Mode): Specifies the algorithm (e.g., MD5, SHA1, NTLM).
- `-a` (Attack Mode): Specifies *how* Hashcat should attempt to guess the plaintext.

**Key Hash Modes (`-m`):**
- `0`: MD5
- `100`: SHA1
- `1000`: NTLM (Windows passwords)
- `1800`: SHA-512 (Linux `/etc/shadow` passwords)

**Key Attack Modes (`-a`):**
- `-a 0` (Straight / Dictionary): Tries every word in a provided wordlist (like `rockyou.txt`).
- `-a 3` (Brute-force / Mask): Generates candidates based on a specific character pattern (a "mask").

**Step 5 — Executing Hashcat Attacks**

I took the MD5 hash generated earlier (`5d41402abc4b2a76b9719d911017c592`) and used Hashcat to crack it back to "hello".

**Example 1: Mask Attack (`-a 3`)**
Because I knew the target was exactly 5 characters long, I used a mask attack. 

```bash
hashcat -m 0 -a 3 5d41402abc4b2a76b9719d911017c592 ?a?a?a?a?a
```
*Breakdown of the command:*
- `-m 0`: Tells Hashcat the target is an MD5 hash.
- `-a 3`: Tells Hashcat to use a mask/brute-force attack.
- `5d41402...`: The target hash.
- `?a?a?a?a?a`: The mask. `?a` stands for "all characters" (lowercase, uppercase, digits, and symbols). Five `?a`s tell Hashcat to try every possible 5-character combination.

Hashcat successfully reversed the hash to `"hello"` in a fraction of a second.

**Understanding Masks:**
Masks allow for highly optimized brute-forcing by reducing the keyspace. If you know a company's password policy requires a capital letter followed by 5 lowercase letters and 2 digits (e.g., `Password12`), you can define exactly that:
- `?l` = lowercase letters
- `?u` = uppercase letters
- `?d` = digits
- `?s` = special characters
- `?a` = all of the above
- *Example mask for the policy:* `?u?l?l?l?l?l?d?d`

**Example 2: Dictionary Attack (`-a 0`)**
If the password length or structure is unknown, a dictionary attack is more efficient.

```bash
hashcat -m 0 -a 0 target_hashes.txt /usr/share/wordlists/rockyou.txt
```
This instructs Hashcat to hash every single word in `rockyou.txt` using MD5 (`-m 0`) and compare the result against the hashes stored in `target_hashes.txt`. 

**Step 6 — Expanding to modern hashes (SHA-256)**

MD5 is largely obsolete. To understand a modern workflow, I generated and cracked a SHA-256 hash.

```bash
# 1. Generate the SHA-256 hash
echo -n "admin123" | sha256sum
# Output: 240be518fabd2724ddb6f04eeb1da5967448d7e831c08c8fa822809f74c720a9  -

# 2. Identify the hash (if unknown)
hashid -m 240be518fabd2724ddb6f04eeb1da5967448d7e831c08c8fa822809f74c720a9
# Output: SHA-256 [Hashcat Mode: 1400]

# 3. Crack it using a dictionary attack
hashcat -m 1400 -a 0 240be518fabd2724ddb6f04eeb1da5967448d7e831c08c8fa822809f74c720a9 /usr/share/wordlists/rockyou.txt
```
Because `admin123` is a common password, the dictionary attack cracked the SHA-256 hash instantly.

## Challenge Encountered

> **Obstacle:** When attempting to crack a hash, Hashcat immediately exited with the message `Exhausted` without finding the password, even though I knew the password was in the wordlist.

**Root cause:** I had previously cracked the exact same hash during testing. Hashcat stores successfully cracked hashes in a local potfile (`~/.local/share/hashcat/hashcat.potfile`). When it encounters a hash it has already solved, it skips it to save time, outputting nothing unless explicitly asked.

---

## Resolution

1. Added the `--show` flag to the command to display hashes that were already cracked in the potfile:
   `hashcat -m 0 --show 5d41402abc4b2a76b9719d911017c592`
2. Alternatively, to force Hashcat to ignore the potfile and crack the hash from scratch, I learned to use the `--potfile-disable` flag.

> **Key realization:** Security tools are designed for efficiency. Understanding where a tool caches its data (like the potfile) is essential for troubleshooting when a tool behaves unexpectedly during repeated testing.

---

## Key Takeaways

- **MD5 is broken:** While fine for simple checksums, MD5 is computationally trivial to crack with modern hardware and should never be used for password storage.
- **Masks reduce keyspace:** Brute-forcing blindly (`?a?a?a?a?a?a?a?a`) takes exponentially longer than using intelligence to restrict the mask (`?u?l?l?l?l?l?d?d`). Efficient password cracking is about math and probability, not just raw compute power.
- **The newline trap:** The difference between `hello` and `hello\n` represents a completely different hash. Precise data handling is mandatory in cryptography.
